# 协作流程

## 对 agent 的期望

1. 先判断本次改动属于：CLI、探测、评估、初始化、升级、模板、预设、运维工具（doctor/export/stats）、插件。
2. 只读取必要文件，禁止全仓扫描。
3. 优先做小步可验证改动。
4. 把重复错误前移成脚本检查或模板字段。

## 对人类评审的期望

1. 先看生成结果是否合理，再看代码风格。
2. 模板新增或修改时，确认是否仍然通用。
3. 看见重复口头提醒时，推动把它们变成脚本检查或模板字段。

## 命令规范

- 小改动：至少运行 `make check`。
- 改评估、升级、初始化逻辑或模板：运行 `make ci`。
- 改 CLI 命令：运行 `make ci`，并实际跑一次对应命令验证。
- 改文档但不改代码：运行 `make check`。

## 提交前检查

1. 改动目标是否能在一句话里描述。
2. 是否有对应测试（当前 134 个，新功能必须补测试）。
3. 是否有文档同步（product.md、architecture.md、runbook.md）。
4. 是否新增了隐藏命令或隐式约束。
5. 如果动了 CLI 或命令入口，是否同步更新了 `docs/runbook.md`。
6. 如果动了模板，是否验证了生成结果。
7. 如果动了评估逻辑，是否验证了评估输出。
8. 如果动了升级逻辑，是否验证了备份和覆盖行为。
9. 如果新增了模块，是否加到 `scripts/check_repo.py` 的 REQUIRED_FILES。
10. 模块是否超过 280 行限制。
