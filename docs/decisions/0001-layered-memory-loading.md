# ADR 0001：分层记忆加载策略

- **状态**：Proposed
- **日期**：2026-04-12
- **决策者**：LFG #10 任务
- **相关 Issue**：GitHub #10（absorbed from MemPalace）

## 背景和问题

`.agent-harness/` 下的 `lessons.md` 和 `task-log.md` 会随项目推进不断增长。当前 `task-lifecycle` 规则指示 Claude 在开始新任务时读取 `lessons.md` 全量 + `task-log.md` 最近 5 条，随文件膨胀将挤占上下文窗口。

MemPalace 项目提出 L0-L3 四层记忆栈：身份+热知识默认加载，温/冷知识按需或显式查询。问题是：**如何把这个思想用最小的代价移植到本脚手架项目？**

## 考虑过的方案

| 方案 | 文件布局 | 机制 | 改动量 | 脚手架适配度 |
|------|---------|------|-------|-------------|
| A. 仅改规则读什么 | 单文件不变 | 规则限流（读最近 N 条） | 小 | 未真正分层 |
| B. 目录分层 | `memory/{L0,L1,L2,L3}/` | 结构化目录 + 主题分区 | 大 | 迁移成本高 |
| **C. Index + 按需展开** | 新增 `memory-index.md`，原文件不变 | 规则读 index，需要深入时用 `/recall` 或 grep | 中 | **最合** |
| D. MemPalace 深度移植 | 同 B + Python `layers.py` | 运行时抽象 + API | 最大 | 过度工程 |

## 决策

**采纳方案 C**：新增一个精华索引文件 `.agent-harness/memory-index.md`，修改 `task-lifecycle` 规则让 Claude 默认只读 index，新增 `/recall` 技能按需搜索 lessons/task-log。

### 关键设计点

1. **保留原文件**：`lessons.md` / `task-log.md` 不拆分、不重命名；git 历史、人工阅读、跨项目迁移都不受影响
2. **index 容量**：最近 10 条教训 + 最近 5 条任务 + 可选主题索引（人工维护）
3. **维护责任**：`/compound` 技能负责写新教训时同步更新 index
4. **升级路径**：
   - `memory-index.md` 列为 `skip` 策略（首次创建，之后保留用户编辑）
   - 老项目可运行 `harness memory rebuild` 从现有 lessons 一次性生成初版 index

## 理由

- **脚手架本质**：本项目是生成器，不是运行时系统——简洁比完备更有价值
- **迁移平滑**：老项目不需要结构迁移，只需创建一个新文件
- **痛点直击**：`task-lifecycle` 规则的读取列表直接受控，AI 默认上下文预算可稳定在 ~200 tokens
- **可演进**：若将来主题索引难以手动维护，可升级到方案 B（按主题分区目录）而不破坏现有数据

## 后果

**正面**：
- 上下文膨胀问题根治
- 改动聚焦（~15 文件以内）
- 保留所有历史文件，零信息损失

**负面**：
- `memory-index.md` 与 `lessons.md` 存在双写风险（靠 `/compound` 规则保证一致性，`/health` 后续可加一致性检查）
- "主题索引"段若无人维护会腐化（可接受，已标为可选）
- 老项目需要主动运行 `rebuild` 才能获得有意义的 index（改为升级时自动 bootstrap 是后续演进项）

## 后续

- 若一致性问题频发 → 加 `/health` 的一致性检查项
- 若 index 继续膨胀 → 升级为主题分区（方案 B）
- 若需跨项目共享记忆 → 考虑 meta 项目类型集成
