# 处理需求讨论记录

你的任务是读取 `notes/` 下的原始讨论记录（多人对话的语音转文字），提取结构化信号，根据项目状态产出 `/process-notes` 或 `/lfg` 可消费的产物。

## 背景与定位

当前项目：`Agent Harness Framework`（cli-tool）

本命令和 `/process-notes` 的分工：

| 命令 | 处理什么 | 噪音水平 |
|------|---------|---------|
| `/process-notes` | 单人已整理的 `notes/` 下 markdown/txt | 低 |
| `/digest-meeting`（本命令） | **多人对话、有分歧、有决策过程**的原始讨论记录 | 高 |

适用场景：
- **新项目 idea 讨论**：团队头脑风暴的录音 → 第一版产品需求
- **迭代评审讨论**：产品经理和研发评审的录音 → 下一迭代任务

## 输入格式兼容（启发式检测）

本命令按**宽进严出**原则兼容 4 种输入格式：

| 格式 | 特征 | 归一化处理 |
|------|------|-----------|
| **飞书妙记导出** | 有"自动生成"、章节标题、固定 header/footer | 提取章节结构，跳过模板装饰 |
| **带说话人 + 时间戳** | `[00:05:23] 张三: ...` 或 `张三 (00:05): ...` | 提取参与者列表 + 按说话人分段，时间戳作分段锚点 |
| **仅带说话人** | 行首 `张三：` 或 `Speaker 1:` | 同上（无时间戳） |
| **纯文本** | 无说话人无时间戳的段落 | 按空行分段，视为单人整理的讨论记录 |

**核心原则**：不确定格式就**展示解析结果让用户校对**，不硬编码 parser。遇到混合格式按行启发式判定。

## 执行步骤

### 第 1 步：读取 `notes/` 下的原始讨论记录

1. 扫描 `notes/` 目录，收集所有 `.md` 和 `.txt` 文件
2. **跳过**以下文件：
   - `notes/digested/` 子目录下（本命令的产物，避免二次处理）
   - 文件头部已有 `<!-- processed: YYYY-MM-DD -->` 标记（除非用户传 `--force`）
3. 目录不存在或全部被跳过 → 报错："未找到待处理的讨论记录"并退出，**不**创建空产物

### 第 2 步：检测模式（init / iterate）

读取 `docs/product.md`，判断"功能列表"段是否有实质内容：

- **init 模式**（功能列表为空、全是 `- [ ] 待补充`、或只有模板占位文字）
- **iterate 模式**（功能列表有非占位的 checkbox 条目，说明产品已经有一版需求）

**判定用**：`grep -A 30 "功能列表\|功能点\|Features" docs/product.md` 看是否有实质内容。不确定时**展示判定理由**让用户确认。

用户可在展示摘要阶段手动覆盖模式选择。

### 第 3 步：提取 6 类信号

从讨论记录中按以下 6 类整理：

| 类别 | 含义 | 注意 |
|------|------|------|
| **决策** (Decisions) | 讨论达成的共识，含分歧→讨论→结论的过程 | 保留分歧和选择理由，不只记结果 |
| **需求** (Requirements) | 功能/行为/用户故事描述 | 口语化→简洁文档语言，合并同类项 |
| **约束** (Constraints) | 技术/时间/合规/资源限制 | 区分"硬约束"（必须）和"偏好"（可改） |
| **待办** (Action Items) | 谁在什么时候做什么 | 保留责任人和时间节点 |
| **开放问题** (Open Questions) | 讨论了但**未达成结论**的话题 | 这是 iterate 下期待办的种子 |
| **参与者** (Participants) | 谁说了什么、谁是决策者 | 从说话人标签提取，无标签时标注"参与者未识别" |

**每条提取项必须标注来源**：
- `（来自：讨论第 N 段原话）` — 直接引用或近乎直接引用
- `（来自：AI 综合推断）` — 基于讨论上下文的推断

**提取原则**：
- 合并同类项（散落多处的同一话题归到一起）
- 忽略噪音（语气词、重复、无信息量的口头禅）
- 存疑标注（含糊/矛盾的用 `<!-- 待确认：... -->` 标注）

### 第 4 步：meta 项目检测（自动委托）

检查当前项目是否是 meta 项目：

```bash
test -f services/registry.yaml && test -d business/domains
```

如果是 meta 项目：
- **🔴 停下来告诉用户**：这是 meta 项目，建议用 `/meta-create-task` 从会议纪要生成跨服务任务，而非本命令。
- 询问："要改走 `/meta-create-task` 吗？" 用户确认 → 委托（告知用户手动运行）；拒绝 → 继续本命令（小众场景，用户知道自己在做什么）。

非 meta 项目 → 继续下一步。

### 第 5 步：展示摘要（🔴 用户确认）

**在写入任何文件之前**，向用户展示提取的 6 类信号摘要：

```
## 讨论记录摘要

**文件**：notes/<原始文件名>
**检测格式**：<飞书妙记 / 带说话人 + 时间戳 / ... / 纯文本>
**检测模式**：<init / iterate>（依据：<判定说明>）
**参与者**：<A、B、C>（或"未识别"）

### 决策
1. <内容>（来自：讨论第 3 段）
   - 分歧点：<A 主张 X，B 主张 Y>
   - 最终：<选 X，理由是 ...>

### 需求
1. [P0] <一句话描述>（来自：讨论第 5 段原话）
2. [P1] <...>（来自：AI 综合推断）

### 约束
- <硬约束 1>（来自：讨论第 2 段）

### 待办
- <@张三 本周出 API 设计>（来自：讨论第 8 段）

### 开放问题
- <支付回调的幂等方案未定>（来自：讨论第 6 段）

### 建议下一步
- init 模式：将上述写入 notes/digested/meeting-YYYY-MM-DD.md，然后运行 /process-notes 填充 docs/product.md
- iterate 模式：将上述写入 current-task.md（任务目标 + 假设 + 验收标准），然后运行 /lfg
```

询问用户：
- "确认写入？" / "要调整哪些内容？" / "要改模式吗？"

用户拒绝 → 不写入任何文件，退出。

### 第 6 步：写入（按模式分叉）

#### init 模式

1. 产物路径：`notes/digested/meeting-YYYY-MM-DD-<简短主题>.md`（目录不存在则创建）
2. 产物格式：`/process-notes` 兼容的结构化笔记，包含：
   - 产品定位（从"需求"类提取）
   - 功能列表（从"需求"类转 `- [ ]` checkbox）
   - 完成标准（从"决策"+"约束"推）
   - 技术选型（从"决策"中的架构类）
   - 开放问题（原样保留，作为下阶段任务种子）
3. **提示用户下一步**：
   ```
   ✅ 已产出 notes/digested/meeting-YYYY-MM-DD-<主题>.md
   下一步：运行 /process-notes 把内容填充到 docs/product.md 等框架文档
   ```

#### iterate 模式

1. 产物路径：`.agent-harness/current-task.md`（覆盖写入）
2. 产物格式（对应 task-lifecycle 规则要求）：
   ```markdown
   # Current Task

   ## 任务目标
   <从"需求"和"决策"综合提取，一句话目标>

   ## 假设清单
   - 假设：... | 依据：讨论第 N 段
   - 假设：... | 依据：AI 综合推断（待确认）

   ## 验收标准
   - [ ] <从"需求"转可验证条目>
   - [ ] ...

   ## 讨论来源
   - 原始记录：notes/<文件名>
   - 处理时间：YYYY-MM-DD
   - 参与者：<...>
   - 开放问题：<未解决的项>
   ```
3. **如果"决策"中涉及架构变更**（技术栈切换、新模式引入等）→ 额外提示用户先跑 `/adr` 记录决策
4. **提示用户下一步**：
   ```
   ✅ 已产出 current-task.md
   下一步：运行 /lfg 开始开发（/lfg 会根据复杂度自动决定是否需要 /spec）
   ```

### 第 7 步：标记已处理

在**每个已处理**的原始文件头部插入：

```html
<!-- processed: YYYY-MM-DD by /digest-meeting -> notes/digested/<产物> -->
```

**绝不修改原始文件内容**，只加头部标记，供下次运行时跳过。

---

## 注意事项

- **原始文件永不删除或改写**，一手资料需长期保留以备回溯
- **产物放 `notes/digested/`**，与原始记录物理分离
- 写入完成后运行 `make check` 验证项目状态不受影响
- 如遇文件编码异常（非 UTF-8）→ 清晰报错，不猜测解码
- 如遇讨论过短（< 100 字）或全是闲聊 → 告知用户"未提取到有效信号"并退出，不硬凑产物

## 与 task-lifecycle 的关系

iterate 模式产出 current-task.md 后，task-lifecycle 规则第 2 步（展示并等待用户确认）仍然生效——用户看到摘要确认后再跑 `/lfg`。本命令**不**绕过确认步骤。
