# 接收代码审查

收到代码审查反馈后，按照六步流程逐条处理。独立思考，拒绝表演性认同。

## 六步处理流程

### 1. 完整阅读（READ）

- 一次性读完所有反馈，不要读一条就改一条
- 标记每条反馈的类型：bug、设计、风格、性能、安全

### 2. 理解复述（UNDERSTAND）

- 用自己的话复述每条反馈的核心观点
- 如果无法准确复述，说明你没有真正理解——**停下来请求澄清**
- 多条反馈中有任何一条不清楚，都必须先澄清再行动

### 3. 代码库验证（VERIFY）

- 在 Agent Harness Framework 代码库中验证反馈提到的问题是否确实存在
- 检查相关文件和调用链
- 运行 `python -m unittest discover -s tests -v` 确认当前状态
- 运行 `python scripts/check_repo.py` 检查代码规范

### 4. 评估合理性（EVALUATE）

对每条反馈独立判断：

- 这个修改是否真的能改善代码？
- 是否符合 Agent Harness Framework 的架构设计？
- **YAGNI 检查**：在代码库中搜索实际使用情况，确认建议的功能确实有需求
- 是否与现有实现存在冲突？

### 5. 回应（RESPOND）

对每条反馈给出明确回应：

**接受时**：
- 说"已修复"+ 具体改了什么
- 不要说"你说得太对了！"或"感谢指出！"——这是表演性认同，禁止使用

**拒绝时**：
- 给出具体的技术理由
- 引用代码库中的证据

### 6. 逐条实施（IMPLEMENT）

- 一次只改一条反馈对应的内容
- 每改完一条运行 `python -m unittest discover -s tests -v`
- 确认不影响其他已通过的修复

## 禁止行为

- **表演性认同**：如"你说得完全正确！"、"非常感谢你的洞察！"、"这是个绝妙的建议！"
- **批量修改**：同时改多条反馈的内容，导致无法定位回归
- **未经验证就接受**：不检查代码库就同意修改
- **未经思考就拒绝**：不给理由就否定反馈

## 合理的反驳场景

以下情况你应该据理力争，而非盲目接受：

| 场景 | 说明 |
|------|------|
| **破坏功能** | 建议的修改会导致现有功能失效 |
| **缺少上下文** | 审查者不了解某个设计决策的背景 |
| **违反 YAGNI** | 建议添加的功能在代码库中没有实际使用场景 |
| **技术错误** | 审查意见本身存在事实性错误 |
| **架构冲突** | 建议与 Agent Harness Framework 的整体架构方向矛盾 |

## 回应格式

```markdown
### 反馈 [编号]：[简述]
- 理解：[用自己的话复述]
- 验证：[代码库中的发现]
- 判断：接受 / 拒绝 / 需要讨论
- 行动：[具体做了什么 / 为什么不做]
- 测试：python -m unittest discover -s tests -v 通过
```

## 核心原则

- 独立思考比快速响应更重要
- 每条反馈都值得认真验证
- 拒绝是合理的，但必须有理由
- 修复的确认方式是"已修复 + 变更内容"，不是感谢
- 一次改一条，保持可追溯性
- 项目类型：cli-tool，语言：python
