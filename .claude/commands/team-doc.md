# Team-Doc — 文档场景编队

铁律：**文档不是顺手写的副产品。模糊需求不动笔，引用未核实不定稿。**

> **场景化预制流水线（吸收自 [Donchitos/Claude-Code-Game-Studios](https://github.com/Donchitos/Claude-Code-Game-Studios) `/team-*` 模式，Issue #55）**：
>
> 当任务是「写一份文档」（标书 / 规范 / 白皮书 / 报告）时，按本流水线串联 `/outline-doc` → `/draft-doc` → `/review-doc` → `/finalize-doc`。
> 跟 `/lfg-doc` 的关系：`/lfg-doc` 是文档场景的**完整流水线**（含 ideate / brainstorm / spec 前置阶段）；`/team-doc` 是**只覆盖大纲到定稿那一段**的局部编队，跳过前置探索，适合**已有规格的文档项目快速推进**。
> 走 D 方案：4 个 team-* skill 各自独立完整写一遍流水线，不抽共享 base（教训来自 lessons.md 2026-04-27）。

当前项目：`Agent Harness Framework`（cli-tool / python）

---

## 参数检查

如果 `$ARGUMENTS` 为空，输出：

> 用法：`/team-doc [文档目标 | 规格文档路径]` — 从大纲到定稿的文档编队。

> 示例：`/team-doc docs/superpowers/specs/2026-04-27-tender-spec.md`、`/team-doc 写一份 API 安全白皮书 5000 字`

然后**立即停止**，不进入流水线。

如果 `$ARGUMENTS` 是规格文档路径 → 直接读取作为输入；如果是文字描述 → 先确认有无字数下限 / 目标读者 / R-ID，缺则建议先跑 `/lfg-doc` 走完整流水线。

---

## 适用场景

- 写文档（标书 / 规范 / 白皮书 / 报告 / 教程）
- 已有规格 / 验收标准 / 字数下限
- 跳过 `/lfg-doc` 前置探索阶段（ideate / brainstorm / spec 已完成）

**不适用**：
- 完全新主题，没思路 → 走 `/lfg-doc` 完整流水线（含 ideate / brainstorm）
- 单段文案 / 短说明 → 直接写
- 写代码 → 走 `/team-implement`

## 编队组成

| 阶段 | 调用 skill | 产出 |
|---|---|---|
| 大纲 | `/outline-doc` | 章节树 + 字数估算 + R-ID 覆盖 + 引用占位 |
| 草稿 | `/draft-doc` | 两段法（outline-pass + draft-pass）产出可读全文 |
| 评审 | `/review-doc` | 4 人格并行（准确性 / 可读性 / 术语统一 / 完整性） |
| 定稿 | `/finalize-doc` | 8 项必检（无占位符 / R-ID 全覆盖 / 术语一致 / 引用核验等）+ 最终文件 |

## 流水线

### 阶段 1：大纲

调用 `/outline-doc`，输入是 `$ARGUMENTS`（spec 或描述）。

`/outline-doc` 产出：
- 章节树（多级 heading）
- 每章字数估算（合计 ≥ 字数下限）
- R-ID 覆盖（每条 R-ID 至少出现在一个章节）
- 引用占位（`[ref:1]`、`[ref:2]` …，留待评审阶段核实）

**🔴 等用户确认大纲**。大纲不通过不进草稿——大纲返工成本远低于全文返工。

### 阶段 2：草稿

调用 `/draft-doc`，输入是阶段 1 的大纲。

两段法：
1. **outline-pass**：每章先写「主张句 + 论据骨架」，不展开细节
2. **draft-pass**：基于 outline-pass 展开成段落，字数填到估算

每章独立 commit，便于回退到具体章节。

**反偷懒门禁 #5（不确定性输出）**：草稿中所有事实性断言必须可归类为：
- **observed**：有引用 / 数据 / 文件路径支撑 → `[ref:N]` 标注
- **uncertain**：凭记忆或推测 → 挂 `⚠️ 待核实` + 指明核实方式

不允许用「大概率」「应该是」「据我所知」规避——这些词仍在做确定性陈述只是把责任推给读者。

### 阶段 3：评审

调用 `/review-doc`，输入是阶段 2 的全文草稿。

4 人格并行评审（独立 SubAgent，反偷懒门禁 #2）：
- **准确性**：事实是否经核实？引用是否真实？数据是否对得上？
- **可读性**：长句 / 黑话 / 重复 / 段落连贯性
- **术语统一**：同一概念是否全文一致？大小写归一？
- **完整性**：R-ID 全覆盖？字数达下限？无 TODO / TBD？

**🔴 准确性发现 P0（事实错误 / 引用伪造） → 立刻停下**回阶段 2 修。其他级别按 P0/P1/P2 分级。

### 阶段 4：定稿

调用 `/finalize-doc`，输入是阶段 3 通过的草稿。

`/finalize-doc` 8 项必检：
1. 无 TODO / TBD / 待补充 / FIXME
2. R-ID 全覆盖（无 missed）
3. 术语一致（按术语表归一）
4. 引用核验（所有 `[ref:N]` 都已替换为真实出处）
5. 字数达标
6. 标点符合 CLAUDE.md 中文规范（直角引号 / 中英文留白 / 不用感叹号）
7. 无禁用黑话词（赋能 / 抓手 / 闭环 / 沉淀 ……）
8. 无第二人称（你 / 您 / 同学）

任一未通过 → 回阶段 2 或阶段 3 修复。

**`/finalize-doc` 不调用 `/git-commit` / `/finish-branch`**——文档定稿后由用户决定如何归档（写入文档管理系统 / 发邮件 / 提 PR），编队不替用户做这个决策。

### 阶段 5：签收

收集四阶段产物，输出 verdict：

```
## Team-Doc 编队报告

文档目标：<$ARGUMENTS>

### 阶段产出
- 大纲：<file>（章节数：N，字数估算：M）
- 草稿：<file>（实际字数：M'，commit 数：N）
- 评审：4 人格 verdict（准确性 / 可读性 / 术语 / 完整性 各自结论）
- 定稿：<file>（8 项必检全通过）

### 关键发现
- 准确性修正：N 处事实错误已修
- 术语统一：N 处归一
- 引用核验：N 个 [ref:N] 已替换为真实出处

### 后续动作（由用户决定）
- 提交到云文档（飞书 / Notion / Confluence）
- 发邮件给评审方
- 提交到代码仓库（如是开源项目文档）
```

## 错误恢复协议

任一阶段返回 BLOCKED / 错误 / 无法完成时：

1. **立即上报**：`[<阶段>]: BLOCKED — <理由>`
2. **始终产出部分报告**
3. **三选一让用户决策**：
   - **跳过当前阶段**：仅适用于评审阶段（如某个人格视角失败但其他通过）
   - **回退重写**：大纲不行回阶段 1，草稿不行回阶段 2
   - **停下解决**：等用户提供新信息（如缺引用源）

常见阻塞：
- 阶段 1：R-ID 无法全覆盖（章节装不下） → 字数下限太低 / 规格冗余，回 `/team-spec` 重审
- 阶段 2：草稿大量 `⚠️ 待核实` → 信息源不足，先做调研 / 跑 `/source-verify` 核实再续写
- 阶段 3：准确性 P0 反复出现 → 信息源本身不可靠，需换源
- 阶段 4：8 项必检某项反复未通过 → 该项有结构性问题，需重新审视全文

## 文件写入约定

- 本编队**只**写文档文件，**不**改业务代码
- 写入路径白名单：
  - `docs/`（项目内文档）
  - `notes/digested/`（已结构化的讨论记录）
  - `.agent-harness/current-task.md`（进度更新）
  - 用户指定的其他文档路径
- 禁止写入：`src/`、`tests/`、`.claude/rules/`（不是文档场景该碰的位置）

---

## 输出

**verdict 三态**：
- **COMPLETE** — 4 阶段全跑完，定稿 8 项必检通过
- **PARTIAL** — 部分阶段完成（如已交大纲 + 草稿但未评审定稿）
- **BLOCKED** — 在阶段 X 阻塞，需用户介入

## 下一步建议
