# Git 提交助手

引导你完成高质量的 git 提交，确保每次提交原子化、规范化、安全。

项目：`Agent Harness Framework`（cli-tool / python）

## 铁律

- **绝不使用 `git add .` 或 `git add -A`**——可能暂存密钥、临时文件等不该提交的内容
- **绝不使用 `--no-verify` 跳过钩子**——钩子存在是有原因的
- **绝不在未被明确要求时使用 `--amend`**——amend 会修改已有提交，可能丢失历史
- **提交消息说 WHY，不说 WHAT**——diff 已经展示了 WHAT

## 第 1 步：上下文收集

运行 `git status`、`git diff --stat`、`git diff --cached --stat`、`git branch --show-current`、`git log --oneline -5`，掌握当前分支、暂存状态和最近提交风格。

## 第 2 步：规范检测

按优先级检测提交消息规范：

1. **项目文档约定**：检查 `CONTRIBUTING.md`、`AGENTS.md` 中是否定义了提交格式
2. **历史模式**：分析最近 5 条提交消息的格式（是否使用 conventional commits、是否有 scope 等）
3. **兜底**：如果以上均无明确规范，使用 Conventional Commits 格式：`type(scope): subject`

常见 type：`feat`、`fix`、`refactor`、`docs`、`test`、`chore`、`perf`、`ci`

## 第 3 步：逻辑分组

审查 diff 内容，判断变更是否涉及多个不相关的关注点：

- 如果变更可以逻辑拆分为独立的提交（例如：一个 bug 修复 + 一个新功能），**建议拆分**
- 每个提交应当是原子化的——单独 revert 不会破坏其他功能
- 向用户说明拆分建议和理由，等待确认后再继续

如果变更本身就是单一关注点，直接进入下一步。

## 第 4 步：智能暂存

使用 `git add path/to/file` 逐文件添加，每个路径明确写出。如果用户未指定提交哪些文件，根据逻辑分组的结果选择本次应包含的文件。

## 第 5 步：安全检查

在提交前完成以下检查：

### 分支检查

- **main / master / 默认分支**：警告用户正在主分支上提交，建议先创建功能分支：
  ```
  ⚠ 当前在主分支上。建议先创建功能分支：git checkout -b feature/xxx
  ```
- **detached HEAD**：警告并解释选项（创建新分支保存工作，或回到已有分支）

### 敏感文件检查

扫描已暂存文件，检查是否包含：

- `.env`、`.env.*` 文件
- 包含 `credential`、`secret`、`key`、`token`、`password` 的文件名
- `id_rsa`、`*.pem`、`*.key` 等私钥文件

如果发现敏感文件：**立即阻止提交**，取消暂存该文件，并警告用户。

## 第 6 步：消息撰写

按照第 2 步检测到的规范撰写提交消息：

格式：`type(scope): 主题行`（不超过 50 字符），可选正文解释 WHY（每行不超过 72 字符）。主题行用祈使语气、不加句号，正文与主题行之间空一行。

## 第 7 步：执行提交

执行 `git commit -m "<主题行>"` （有正文时加 `-m "<正文>"`）。

提交成功后运行 `git log --oneline -1` 确认，向用户报告提交哈希、主题行、分支名和变更文件数。

## 提交失败处理

- **pre-commit 钩子失败**：阅读钩子输出，修复问题，重新暂存，创建**新提交**（不要 amend）
- **commit-msg 钩子失败**：根据钩子要求调整消息格式，重新提交
- **冲突或其他错误**：向用户报告完整错误信息，等待指示

## 一句话总结

**逐文件暂存 → 安全检查 → 规范消息 → 确认结果。不走捷径。**
