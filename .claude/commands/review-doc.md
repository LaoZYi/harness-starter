# 文档评审（Review-Doc）

铁律：**4 个评审人格独立 SubAgent 并行跑，逐条回应不批量 dismiss。**

写文档的评审与代码评审（`/multi-review`）同源思想，但角度替换为文档专用四维：**准确性 / 可读性 / 术语统一 / 完整性**。

---

## 何时使用

- `/draft-doc` 完成后、`/finalize-doc` 之前
- 文档大改后想验质量
- 多人协作场景下交叉审稿

不适用：
- 还在 outline 阶段（用 `/spec` 校验 R-ID 即可）
- 单页备忘 / 公告 / 通知（杀鸡用牛刀）

---

## 前置条件

1. `outline.md` 存在
2. `spec.md` 存在（含 R-ID）
3. 草稿文档已完成（`/draft-doc` 全部章节都展开过）
4. 草稿无 TODO / TBD / 待补充字面量（grep 验证）

下游消费者门禁：

```bash
DRAFT="${DRAFT_PATH:-draft.md}"
test -f "$DRAFT" && test -s "$DRAFT" || { echo "draft missing"; exit 1; }
grep -E "TODO|TBD|待补充" "$DRAFT" && { echo "draft 含未填占位，回 /draft-doc"; exit 1; }
```

---

## 4 个评审人格（独立 SubAgent，门禁 2 上下文隔离）

每个人格 spawn 独立 SubAgent，**不共享上下文**——避免长上下文污染降低评审深度。

### 人格 1：准确性审查员（Accuracy Reviewer）

**视角**：我是这个文档的事实核查员。我只关心"这话是不是真的"。

**逐项检查**：

- 每个数字 / 百分比 / 日期是否有可追溯出处？（`[data:X]` `[ref:Y]` 占位是否都有解析草案？）
- 每个引用是否真实存在？（`[ref:client-survey-2026-q1]` → 这个调研真有？）
- 专有名词大小写 / 拼写是否正确？（`HTTP`、`OAuth 2.0`、`Kubernetes`、`Anthropic` —— 不是 http / oauth2 / k8s / anthropic）
- 强断言是否有依据？（"行业领先" / "首创" / "唯一" 这种话谁背书？）
- 时间表述是否清晰？（"近期"、"很快"、"上周" 都是模糊的——评审里要标）

**输出**：P0/P1/P2 + 可定位行号

```markdown
## 准确性审查报告

P0（事实错误，必须改）
- L42 "占有率 35%" 无 [data:X] 占位也无明示来源 → 加来源或删
- L88 "GPT-5 发布于 2025-09" 错；GPT-5 发布于 2025-12 → 改

P1（可疑，需核实）
- L120 "客户满意度提升 40%" → 与哪个基准比？时间窗？

P2（建议）
- L15 "近期" → 改成具体日期范围
```

### 人格 2：可读性审查员（Readability Reviewer）

**视角**：我是这个文档的目标读者代理。我只关心"读起来顺不顺、累不累"。

**逐项检查**：

- 段落长短是否均衡？（连续 3 个 12+ 行段落 = 累）
- 长难句是否过载？（单句 ≥ 50 字必拆）
- 术语首次出现是否有定义？
- 章节衔接是否自然？（每章开头有没有承接前章的过渡句？）
- 一级标题之间逻辑跳跃大吗？

**输出**：P0/P1/P2 + 可定位行号

### 人格 3：术语统一审查员（Terminology Reviewer）

**视角**：我是这个文档的术语索引。我只关心"同一概念有没有用同一词"。

**逐项检查**：

- 同义词是否在全文用同一种表达？（"用户" vs "客户" vs "使用者" → 选一个）
- 大小写归一是否一致？（HTTP / http / Http → 全文必须统一）
- 缩写首次出现有没有展开？（"AI（Artificial Intelligence）" 第一次 → 后续可直接 AI）
- 数字格式统一吗？（5,000 / 5000 / 5K → 选一种）
- 标点统一？（中文用「」还是""？项目 CLAUDE.md 说用「」）

**输出**：grep 出所有变体 + 建议规范

```markdown
## 术语统一审查报告

冲突 1（必须统一）：
- "用户"出现 12 次（L3, L15, L22, ...）
- "客户"出现 8 次（L41, L67, ...）
- "使用者"出现 3 次（L99, ...）
- 建议：评委 / 客户语境用"客户"，技术语境用"用户"，"使用者"全替换

冲突 2：
- HTTP 出现 5 次，http 出现 2 次（L88, L120）
- 项目 CLAUDE.md 规范 → 统一 HTTP
```

### 人格 4：完整性审查员（Completeness Reviewer）

**视角**：我对照规格 R-ID 一条一条核——每条 R-ID 是不是真在文档里讲到了？

**逐项检查**：

- spec.md 中的 R-ID 是否每条都在文档中可定位？
- outline.md 中标的 R-ID 覆盖与文档实际覆盖是否一致？
- 章节定位句（outline 写的）是否真的兑现？
- 关键论点是否有论据支撑？

**输出**：R-ID 映射表 + missing 清单

```markdown
## 完整性审查报告

R-ID 覆盖矩阵：
| R-ID | 描述 | 文档章节 | 状态 |
|---|---|---|---|
| R1 | 项目背景 | 1 | ✅ satisfied |
| R2 | 技术架构 | 2.1 | ✅ |
| R3 | 关键能力 | 2.2 | ⚠️ 浅层（只列名未论证） |
| R4 | 性能指标 | (未找到) | ❌ missed |
| R5 | 团队介绍 | 4 | ✅ |

阻断（P0）：
- R4 在文档中找不到对应章节，必须补
```

## 反偷懒（数量门禁，门禁 1）

**4 个审查员的反馈，必须逐条回应**：

| 状态 | 含义 | 写法 |
|---|---|---|
| ✅ 接受 | 认同问题，已改 | "已改：L42 改为 ..." |
| 🤔 质疑 | 不同意审查员判断 | "保留原文：理由 ..." |
| ⏸️ 推迟 | 同意但本次不改 | "推迟到 v2：理由 ..." |

**禁止**：

- 「整体看着 OK，4 个审查员的细节问题就先不管了」 → 数量门禁不允许批量 dismiss
- 「这条审查员说得不对」不附理由就跳过 → 必须给出"保留原文"的具体证据
- P0 / P1 全部 close 之前不能进 `/finalize-doc`

## 与 `/multi-review` 的区别

| 维度 | `/multi-review`（代码评审） | `/review-doc`（文档评审） |
|---|---|---|
| 评审角度 | 正确性 / 测试 / 可读性 / 安全 / 性能 / 依赖 | 准确性 / 可读性 / 术语统一 / 完整性 |
| 输出位置 | `docs/superpowers/reviews/<topic>-review-<ts>.md` | `docs/superpowers/reviews/doc-<topic>-review-<ts>.md` |
| 数量门禁 | 6 人格 P0/P1/P2 逐条回应 | 4 人格 P0/P1/P2 逐条回应 |
| 上下文隔离（门禁 2）| 独立 SubAgent | 独立 SubAgent |
| R-ID 反向追溯 | 每个 R-ID 必须 satisfied / out-of-scope / missed | 同 |
| 适用场景 | 代码改动 / PR 评审 | 文档草稿评审 / 标书评委预演 |

---

## 完成标准

- [ ] 4 个审查员各自跑完（独立 SubAgent，不共享上下文）
- [ ] 4 份评审报告全部产出（accuracy / readability / terminology / completeness）
- [ ] 每条 P0/P1 都有明确状态（接受 / 质疑 / 推迟）
- [ ] R-ID 覆盖矩阵完整（每条 R-ID 状态明确）
- [ ] P0 / P1 全部 close 才能进 `/finalize-doc`

---

## 反偷懒（扩展门禁 3）

| 借口 | 驳斥 |
|---|---|
| 「我自己审一遍就够了，不用 4 个人格」 | 单一视角的盲区是结构性的——准确性视角不会注意到术语统一，反之亦然。4 人格分工是为了**强制**穷举 |
| 「让 1 个 SubAgent 跑 4 个人格」 | 违反门禁 2 上下文隔离。4 人格在同一上下文下会互相污染——后跑的人格会被前面人格的视角带偏 |
| 「这文档只有 1000 字，不至于走 4 人格」 | 长度不是质量信号。1000 字标书也会被评委按四维评分——你不审，评委审给你看 |
| 「P2 我先不管」 | P2 列入 known-issues 段，不是直接忽略 |
| 「上次评审过了，这次微调不用再走」 | 改了正文必须重审；上次的 P0 close 不代表这次没新 P0 |
| 「跑 SubAgent 太烧 token」 | 评审漏掉的 P0 在外部曝光后，修复成本比几十 k token 大几个数量级 |

## 与下层规则的关系

- **anti-laziness.md 门禁 1（数量门禁）**：4 人格反馈逐条回应，不批量 dismiss
- **anti-laziness.md 门禁 2（上下文隔离）**：4 人格各自独立 SubAgent
- **anti-laziness.md 门禁 5（不确定性输出）**：审查员遇不确定的事实必须挂 ⚠️ 待核实
- **task-lifecycle.md StuckDetector**：同条评审意见连续 3 次修复仍被标 P0/P1 → 触发卡死，停下问用户

---

## 一句话总结

**评审不是"看一遍找毛病"，是"4 个不同角色独立读完同一份文档，把各自的盲区互相补上"。**
