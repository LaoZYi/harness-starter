# 请求代码审查

在关键节点主动请求代码审查，确保变更质量。提供完整的审查上下文，按严重程度处理反馈。

## 必须请求审查的时机

- 任务完成后，提交合并前
- 重大功能的实现完成时
- 准备向 main/master 合并时
- 架构层面的变更
- 涉及安全或数据处理逻辑的修改

## 准备审查请求

### 收集必要信息

1. **实现描述**：用 2-3 句话概括做了什么、为什么这样做
2. **需求/计划引用**：指向原始需求文档或计划中的具体任务
3. **变更范围**：
   - base commit SHA（分支起点）
   - head commit SHA（当前最新）
   - 涉及的文件和模块列表
4. **测试状态**：`make test` 的运行结果
5. **检查状态**：`make check` 的运行结果

### 编写审查摘要

```markdown
## 审查请求：[简要标题]

### 变更概述
[2-3 句话描述变更内容和动机]

### 关联
- 需求/任务：[链接或引用]
- 计划章节：[引用]

### 变更范围
- Base: [commit SHA]
- Head: [commit SHA]
- 修改文件：[数量] 个
- 项目：Agent Harness Framework（cli-tool）
- 语言：python

### 测试验证
- make test：通过/失败
- make check：通过/失败

### 特别关注点
[希望审查者重点关注的区域或决策]

### 已知限制
[当前实现的已知不足或后续改进计划]
```

## 处理审查反馈

### 按严重程度分类处理

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **严重** | 功能错误、安全漏洞、数据丢失风险 | 立即修复，修复后重新请求审查 |
| **重要** | 设计问题、性能隐患、可维护性问题 | 在推进下一步前修复 |
| **次要** | 代码风格、命名建议、文档改进 | 记录为后续任务，不阻塞当前流程 |

### 修复后的验证

1. 针对每个修复运行 `make test`
2. 确认修复没有引入新问题
3. 更新审查摘要中的状态

## 危险信号

以下行为表明审查流程出了问题：

- **跳过审查**：以"改动很小"为由不请求审查
- **忽视严重问题**：收到严重反馈后继续推进
- **重要问题未修复就合并**：将重要级别的问题降级为次要
- **审查摘要不完整**：缺少 commit SHA 或测试结果
- **审查后大幅修改但不重新审查**：修复过程中引入了新的重大变更

## 核心原则

- 审查是质量保障，不是形式流程
- 提供完整上下文是请求者的责任
- 严重和重要问题必须在合并前解决
- 对审查反馈保持开放态度，但也保留合理质疑的权利
