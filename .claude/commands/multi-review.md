# 多视角代码评审

用 6 个并行审查视角对变更进行结构化评审。目标：**在代码合并前发现尽可能多的问题，分类分级后给出处理建议。**

当前项目：`Agent Harness Framework`（cli-tool / python）
测试命令：`python -m unittest discover -s tests -v`
检查命令：`python scripts/check_repo.py`

## 第 1 步：确定评审范围

按优先级确定要评审的代码差异：

1. 用户指定的文件或路径
2. 当前分支相对于基础分支的差异：`git diff $(git merge-base HEAD main)..HEAD`
3. 暂存区的变更：`git diff --cached`
4. 未暂存的变更：`git diff`

如果差异为空，**停止并告知用户没有可评审的内容**。

## 第 2 步：提取变更意图

在评审前先理解变更的目的：

1. 阅读相关的 commit 消息
2. 检查 `.agent-harness/current-task.md` 中的任务描述
3. 检查 `docs/superpowers/specs/` 下是否有对应的计划文档
4. 总结为一句话：这个变更要达到什么目的？

意图不清时仍然评审，但在报告中标注"意图不明，以下发现基于代码推断"。

## 第 3 步：启动 6 个审查员（并行）

使用子代理并行运行以下 6 个审查视角。每个审查员独立产出发现列表。

### 审查员 1：正确性审查员

- 逻辑错误、条件遗漏、边界情况
- 错误处理：是否吞掉异常、是否处理了空值/零值
- 状态管理：竞态条件、状态不一致
- 数据流：输入验证、类型转换、数据丢失

### 审查员 2：测试审查员

- 新代码是否有对应测试
- 现有测试是否覆盖了变更路径
- 缺失的测试场景：边界值、错误路径、并发
- 测试质量：是否测试了行为而非实现
- 运行 `python -m unittest discover -s tests -v` 确认测试通过

### 审查员 3：可维护性审查员

- 命名是否清晰、一致
- 函数/类是否职责单一
- 耦合度：模块间依赖是否合理
- 复杂度：深层嵌套、过长函数、复杂条件
- 可读性：是否需要注释才能理解

### 审查员 4：项目规范审查员

- 是否遵循项目已有的代码风格和模式
- 文件组织、目录结构是否符合惯例
- 命名约定（变量、函数、文件）是否一致
- 运行 `python scripts/check_repo.py` 检查规范合规性
- 导入顺序、格式化是否符合项目配置

### 审查员 5：安全审查员

- 注入风险：SQL、XSS、命令注入、模板注入
- 认证与授权：权限检查是否完整
- 数据暴露：日志中是否包含敏感信息、API 响应是否过度暴露
- OWASP Top 10 相关问题
- 依赖安全：是否引入了已知漏洞的依赖

### 审查员 6：性能审查员

- N+1 查询、批量操作缺失
- 内存泄漏：未清理的监听器、未关闭的连接
- 算法复杂度：是否有明显的性能陷阱
- 资源使用：大文件加载、不必要的计算
- 缓存：是否有重复计算可以缓存

## 第 4 步：合并与去重

收集所有审查员的发现后：

1. 合并指向同一代码位置的相同问题
2. 当多个审查员发现同一问题时，取最高严重级别
3. 按严重级别排序

## 第 5 步：分类与分级

### 严重级别

| 级别 | 含义 | 处理要求 |
|------|------|----------|
| P0 | 阻断问题，必须修复 | 不修复不能合并 |
| P1 | 重要问题 | 强烈建议修复 |
| P2 | 应当修复 | 建议在当前或下个迭代修复 |
| P3 | 改进建议 | 记录，择机处理 |

### 处理分类

| 分类 | 含义 | 操作方式 |
|------|------|----------|
| `safe_auto` | 可自动修复 | 格式化、导入排序等，直接修复 |
| `gated_auto` | 需确认后自动修复 | 展示修复方案，用户确认后执行 |
| `manual` | 需人工判断 | 提供建议，由开发者决定 |
| `advisory` | 仅供参考 | 记录，不需要行动 |

## 第 6 步：输出评审报告

### 报告格式

```
## 评审摘要

变更意图：<一句话描述>
评审范围：<文件数> 个文件，<行数> 行变更
发现总数：P0: N | P1: N | P2: N | P3: N

## P0 — 阻断问题（必须修复）

| 审查员 | 位置 | 发现 | 分类 |
|--------|------|------|------|
| 正确性 | file.py:42 | 空指针未检查 | manual |

## P1 — 重要问题

...（同上格式）

## P2 — 应当修复

...

## P3 — 改进建议

...

## 结论

<PASS / PASS WITH CONDITIONS / FAIL>
```

### 结论标准

- **PASS**：无 P0/P1，P2 不超过 3 个
- **PASS WITH CONDITIONS**：无 P0，P1 均有明确修复方案
- **FAIL**：存在 P0，或 P1 无法在当前迭代修复

## 后续操作

- 如果结论为 FAIL 或 PASS WITH CONDITIONS，提示用户修复后重新评审
- `safe_auto` 类发现可以询问用户是否一键修复
- `gated_auto` 类发现逐个展示修复方案等待确认

## 外部评审集成

如果项目安装了 [codex-plugin-cc](https://github.com/openai/codex-plugin-cc)，可以在本评审完成后运行 `/codex:review` 获取 Codex AI 的独立评审视角，作为交叉验证。两个不同 AI 的评审结果互补，能更全面地发现问题。
