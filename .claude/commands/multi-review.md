# 多视角代码评审

用 6 个并行审查视角对变更进行结构化评审。目标：**在代码合并前发现尽可能多的问题，分类分级后给出处理建议。**

当前项目：`Agent Harness Framework`（cli-tool / python）
测试命令：`make test`
检查命令：`make check`

## 评审模式选择

支持四种模式，通过参数 `--mode` 指定。未指定时默认 `review`。

| 模式 | 机制 | 适用场景 |
|------|------|---------|
| **review**（默认） | 每个审查员独立评审，**不看其他审查员的结论**——避免锚定效应 | 日常评审 |
| **challenge** | 在 review 模式基础上，额外 spawn 一个「挑战者」SubAgent，明确要求其**尝试推翻**所有 P0/P1 结论 | 高风险变更、架构决策、安全相关 |
| **consult** | 所有审查员可看到彼此的结论，综合多角度判断（即现有行为） | 复杂权衡、无明确对错、需要交叉补充 |
| **cross-model** | 跨模型（Claude + Codex + Gemini 等）并行独立评审 → 交叉验证 → 辩论 → 自动收敛。详见文末「跨模型对抗通道」 | 生产 / 架构 / 安全敏感变更；前置需 ≥ 2 个模型 CLI |

**模式与上下文隔离的关系**（Anti-Laziness Gate #2）：
- `review` / `challenge` / `cross-model` 模式**强制**每个审查员（或每个模型）在独立上下文中执行
- `consult` 模式允许审查员共享上下文（现有行为），但仍建议 spawn 独立 SubAgent

## 第 1 步：确定评审范围

按优先级确定要评审的代码差异：

1. 用户指定的文件或路径
2. 当前分支相对于基础分支的差异：`git diff $(git merge-base HEAD main)..HEAD`
3. 暂存区的变更：`git diff --cached`
4. 未暂存的变更：`git diff`

如果差异为空，**停止并告知用户没有可评审的内容**。

## 第 2 步：提取变更意图

在评审前先理解变更的目的：

1. 阅读相关的 commit 消息
2. 检查 `.agent-harness/current-task.md` 中的任务描述
3. 检查 `docs/superpowers/specs/` 下是否有对应的计划文档
4. 总结为一句话：这个变更要达到什么目的？

意图不清时仍然评审，但在报告中标注"意图不明，以下发现基于代码推断"。

## 第 3 步：启动 6 个审查员（并行）

使用子代理并行运行以下 6 个审查视角。每个审查员独立产出发现列表。

### 上下文隔离（Anti-Laziness Gate #2）

每个审查员**必须**在独立 SubAgent 中执行（spawn 新上下文），禁止在主会话中串行完成所有审查。

原因：长上下文污染会不自觉缩减审查范围——后面的审查员受前面审查员结论的锚定效应影响，容易遗漏独立视角才能发现的问题。

> **例外**：轻量通道（`/lfg` 小任务）只启用 2 个审查员时，可在主会话执行。

### 数量门禁（Anti-Laziness Gate #1）

评审完成后，每条 P0/P1 反馈必须在修复阶段逐条回应（`接受` / `质疑` / `推迟`），禁止批量 dismiss。输出汇总行：

```
数量门禁：P0/P1 反馈 N 条，已回应 N/N（接受: X | 质疑: Y | 推迟: Z）
```

### 审查员 1：正确性审查员

- 逻辑错误、条件遗漏、边界情况
- 错误处理：是否吞掉异常、是否处理了空值/零值
- 状态管理：竞态条件、状态不一致
- 数据流：输入验证、类型转换、数据丢失

### 审查员 2：测试审查员

- 新代码是否有对应测试
- 现有测试是否覆盖了变更路径
- 缺失的测试场景：边界值、错误路径、并发
- 测试质量：是否测试了行为而非实现
- 运行 `make test` 确认测试通过

### 审查员 3：可维护性审查员

- 命名是否清晰、一致
- 函数/类是否职责单一
- 耦合度：模块间依赖是否合理
- 复杂度：深层嵌套、过长函数、复杂条件
- 可读性：是否需要注释才能理解

### 审查员 4：项目规范审查员

- 是否遵循项目已有的代码风格和模式
- 文件组织、目录结构是否符合惯例
- 命名约定（变量、函数、文件）是否一致
- 运行 `make check` 检查规范合规性
- 导入顺序、格式化是否符合项目配置

### 审查员 5：安全审查员

- 注入风险：SQL、XSS、命令注入、模板注入
- 认证与授权：权限检查是否完整
- 数据暴露：日志中是否包含敏感信息、API 响应是否过度暴露
- OWASP Top 10 相关问题
- 依赖安全：是否引入了已知漏洞的依赖

### 审查员 6：性能审查员

- N+1 查询、批量操作缺失
- 内存泄漏：未清理的监听器、未关闭的连接
- 算法复杂度：是否有明显的性能陷阱
- 资源使用：大文件加载、不必要的计算
- 缓存：是否有重复计算可以缓存

## 第 3.5 步：大变更集导航（条件触发）

当评审范围超过 **20 个文件**时，在并行评审开始前先生成变更集导航地图：

1. **文件分组**：按模块/目录将变更文件分组，标注每组的核心变更
2. **依赖关系**：标注文件间的调用/引用关系（A 改了接口 → B 是调用方）
3. **变更热点**：标注改动最大的 3-5 个文件为重点审查对象
4. **阅读顺序建议**：给出从上游到下游的推荐审查顺序

导航地图作为上下文传递给每个审查员，帮助他们理解变更全貌。

> 变更文件不超过 20 个时跳过此步骤，直接进入并行评审。

## 第 4 步：评审辩论（Discourse Round）

并行评审完成后，**不直接汇总**，先进行一轮评审者间的交叉验证。

### 辩论流程

1. 将所有审查员的 P0 和 P1 发现汇总为一份**发现清单**
2. 每个审查员审视其他审查员的关键发现，执行以下四种操作之一：

| 操作 | 含义 | 格式 |
|------|------|------|
| **AGREE** | 以自己的视角提供支持性证据 | `AGREE: <发现编号> — <补充证据>` |
| **CHALLENGE** | 质疑假设、推理或严重级别判定 | `CHALLENGE: <发现编号> — <质疑理由>` |
| **CONNECT** | 将不同审查员的发现关联起来 | `CONNECT: <发现A> + <发现B> — <关联说明>` |
| **SURFACE** | 提出辩论中涌现的新关注点 | `SURFACE: <新发现> — <来源于哪些发现的讨论>` |

3. 根据辩论结果调整发现：
   - 被多人 AGREE 的发现 → 信心增强，保持或提升级别
   - 被 CHALLENGE 且无人反驳 → 降级或移除（标注"辩论后降级"）
   - CONNECT 关联的发现 → 合并为一个更全面的发现
   - SURFACE 涌现的新发现 → 加入发现清单并评级

### 辩论输出

```
## 辩论记录

### 共识发现（被确认的问题）
- P0-1: <描述> — 6/6 审查员 AGREE
- P1-3: <描述> — 4/6 AGREE, 正确性+安全审查员补充证据

### 争议发现（被质疑的问题）
- P1-2: <描述> — 测试审查员 CHALLENGE: <理由> → 降级为 P2

### 关联发现（被连接的问题）
- P1-1 + P2-4: <合并描述> — 性能审查员 CONNECT: 同一数据流上的两个问题

### 涌现发现（辩论中新发现的问题）
- P1-NEW: <描述> — SURFACE by 安全审查员: 来源于正确性和性能发现的交叉分析
```

> 如果 P0/P1 发现总数不超过 2 个，辩论轮可简化为快速确认（每个审查员一句话表态），不需要完整辩论流程。

## 第 4.5 步：争议解决循环（Dispute Resolution Loop）

仅在 `challenge` 模式下触发，或 `review` 模式下辩论阶段出现 CHALLENGE 且无法达成共识时触发。

### 触发条件

- `challenge` 模式：挑战者 SubAgent 对任何 P0/P1 结论提出有效反驳（附代码证据）
- `review` / `consult` 模式：辩论阶段同一发现被 2+ 审查员 CHALLENGE 且无人 AGREE

### 循环流程

1. **论述轮**：冲突双方各出具结构化论述：
   ```
   ## 论述：<审查员角色>
   立场：<漏洞存在 / 漏洞不存在 / 严重级别应为 X>
   证据：
   - <file.py:42> — <代码引用 + 分析>
   - <file.py:88> — <代码引用 + 分析>
   反驳对方：<针对对方证据的具体质疑>
   ```

2. **仲裁轮**：spawn 一个独立 SubAgent 作为仲裁者——拿到冲突双方的论述 + 原始代码，但**不被告知谁是"原始结论"谁是"挑战者"**，独立判断。

3. **裁决**：
   - 仲裁者与某方一致 → 采纳该方结论，标注「经争议解决确认」
   - 仲裁者给出第三种结论 → 回到步骤 1，三方论述（第 2 轮）
   - **最多 3 轮**。第 3 轮仍不一致 → 标记 `NEEDS_HUMAN_REVIEW`，争议记录完整保留在评审报告中

### 争议记录格式

```
## 争议解决记录

### 争议 1：<发现编号> — <一句话描述>
- 轮次 1：审查员 A 认为 P0（证据：...），审查员 B CHALLENGE（证据：...）
- 仲裁：仲裁者认为 P1（理由：...）
- 裁决：降级为 P1（经争议解决确认）

### 争议 2：...
- 裁决：NEEDS_HUMAN_REVIEW（3 轮未收敛）
```

**铁律：争议记录不得静默丢弃**——即使最终裁决与原始结论一致，争议过程也要保留在报告中，供人工复查。

## 第 5 步：合并与去重

收集辩论后的发现清单：

1. 合并指向同一代码位置的相同问题
2. 当多个审查员发现同一问题时，取最高严重级别
3. 按严重级别排序
4. 辩论中被降级或移除的发现标注原因

## 第 6 步：分类与分级

### 严重级别

| 级别 | 含义 | 处理要求 |
|------|------|----------|
| P0 | 阻断问题，必须修复 | 不修复不能合并 |
| P1 | 重要问题 | 强烈建议修复 |
| P2 | 应当修复 | 建议在当前或下个迭代修复 |
| P3 | 改进建议 | 记录，择机处理 |

### 处理分类

| 分类 | 含义 | 操作方式 |
|------|------|----------|
| `safe_auto` | 可自动修复 | 格式化、导入排序等，直接修复 |
| `gated_auto` | 需确认后自动修复 | 展示修复方案，用户确认后执行 |
| `manual` | 需人工判断 | 提供建议，由开发者决定 |
| `advisory` | 仅供参考 | 记录，不需要行动 |

## 第 7 步：输出评审报告

### 报告格式

```
## 评审摘要

变更意图：<一句话描述>
评审范围：<文件数> 个文件，<行数> 行变更
发现总数：P0: N | P1: N | P2: N | P3: N
辩论轮：<已执行/已简化/已跳过> — <N 个发现被确认, N 个被降级, N 个新涌现>

## P0 — 阻断问题（必须修复）

| 审查员 | 位置 | 发现 | 分类 |
|--------|------|------|------|
| 正确性 | file.py:42 | 空指针未检查 | manual |

## P1 — 重要问题

...（同上格式）

## P2 — 应当修复

...

## P3 — 改进建议

...

## 结论

<PASS / PASS WITH CONDITIONS / FAIL>
```

### 结论标准

- **PASS**：无 P0/P1，P2 不超过 3 个
- **PASS WITH CONDITIONS**：无 P0，P1 均有明确修复方案
- **FAIL**：存在 P0，或 P1 无法在当前迭代修复

### 静默丢弃检测（Issue #38）

审查员判定某个潜在问题「不是问题」或「不适用」时，**必须给出具体证据**，不能只写「看起来没问题」：

```
- 位置：file.py:42
- 初始判断：疑似 SQL 注入
- 最终判定：不是问题
- 证据：使用了 PreparedStatement 参数化查询（file.py:38），用户输入未直接拼接
```

**禁止**以下模糊判定：
- 「这个应该没问题」（没有证据）
- 「不在本次范围内」（评审范围由 Step 1 确定，已进入范围的不能中途排除）
- 「上下文太复杂，跳过」（复杂正是需要审查的理由）

被判定为「不是问题」的项汇总到评审报告末尾的 **Dismissed 清单**，格式同上。dismissed 数量 ≥ 发现总数的 50% 时输出告警——说明审查可能过于保守。

## 反合理化

当你想跳过或简化评审时，检查这张表。

| 你想说的 | 为什么不行 |
|---------|-----------|
| "改动太小，不值得做 6 人评审" | 小改动可以快速评审，但不应跳过。生产事故常来自"一行代码的修改" |
| "我自己写的代码，我很清楚没问题" | 评审的价值恰恰在于引入你没有的视角。自己看不到自己的盲区 |
| "时间紧迫，先合并再说" | 未经评审就合并的代码，出问题后的修复时间远大于评审时间 |
| "这只是重构，逻辑没变" | 重构最容易引入微妙的行为变化。正确性审查员和测试审查员能抓住这些 |
| "评审发现太多 P2/P3，太碎了" | P2/P3 不阻塞合并，但记录下来能防止技术债积累。忽略它们不会让问题消失 |
| "安全审查没必要，这不是面向用户的代码" | 内部代码的安全漏洞被利用时往往更危险，因为缺少外部监控 |

**铁律不变：在代码合并前发现尽可能多的问题。**

## 后续操作

- 如果结论为 FAIL 或 PASS WITH CONDITIONS，提示用户修复后重新评审
- `safe_auto` 类发现可以询问用户是否一键修复
- `gated_auto` 类发现逐个展示修复方案等待确认

## 跨模型对抗通道（Cross-Model Adversarial CR）

> 来源：腾讯 CDN LEGO 团队实践——单模型 CR 有三大盲区，跨模型对抗能显著提升深层缺陷发现率。

### 为什么需要跨模型

单模型评审（即便多人格 SubAgent）存在同源盲区：

| 盲区 | 表现 | 根因 |
|------|------|------|
| **知识盲区** | 对特定框架 / 模式的理解受训练数据局限 | 不同模型训练语料不同（Claude 更懂 Seastar 异步、Codex 更懂系统调用约定、Gemini 更懂 Google 内部标准） |
| **注意力盲区** | 500+ 行 diff 时后半部分审查质量下降 | 上下文窗口有限，即便同模型多人格也共享此缺陷 |
| **确认偏差** | 发现一类问题后倾向沿同方向继续找 | 模型推理的一致性反而成为单模型的局限 |

### 前置条件

确认本机至少装了 **2 个独立模型的 CLI**：

- Claude Code（本机默认，无需额外安装）
- `codex`（[OpenAI Codex CLI](https://github.com/openai/codex-cli) 或 [codex-plugin-cc](https://github.com/openai/codex-plugin-cc)）
- `gemini` / `gemini-cli`（Google Gemini CLI）
- 其他前沿模型 CLI

仅有 1 个可用 → **降级为本文档前面的 review / challenge 同模型多人格模式**，不要假装在做跨模型。

### 协议：并行独立 → 交叉验证 → 辩论 → 自动收敛

```
Round 1（并行独立）：
  模型 A 独立审查 → 问题集 {a1, a2, a3}
  模型 B 独立审查 → 问题集 {b1, b2, a2}
  模型 C 独立审查 → 问题集 {c1, a1, b1}

  约束：三个模型**不得**看到彼此的中间输出；只看同一份 diff

Round 2（交叉验证）：
  a2 被 A + B 命中 → 高置信度（≥ 2 命中）→ 直接进 P0/P1 候选
  a1 被 A + C 命中 → 高置信度
  b1 被 B + C 命中 → 高置信度
  a3 / c1 单命中 → 进入 Round 3 辩论

Round 3（辩论轮）：
  对每个单命中问题，让另外两个模型评判「同意 / 反对 / 维持」
  同意 ≥ 1 → 升级为高置信；全部反对 → 标注存疑，不入 P0/P1
  争议仍在 → 走第 4.5 步争议解决循环

收敛：任一轮没有新问题 → 自动停，不强制跑满 N 轮
```

### 执行方式（两种）

**方式 A：本项目主会话编排（推荐）**

1. 主会话通过 SubAgent 派一路给 Claude（本项目自身）做 6 人格评审
2. 用 `Bash` 调用本机的 `codex` CLI 生成第二份评审（命令视 CLI 而定）
3. 用 `Bash` 调用本机的 `gemini` CLI 生成第三份评审
4. 主会话收到三份报告后做交叉验证 + 辩论 + 收敛

**方式 B：外部插件分路（降级）**

1. 本 skill 跑完后运行 `/codex:review`（来自 codex-plugin-cc）
2. 对比两份报告，手工做交叉验证
3. 不自动收敛，由用户判断是否补第三份

### 输出格式扩展

在第 7 步的评审报告顶部追加「跨模型置信度」表：

```markdown
## 跨模型置信度

| 问题 ID | Claude | Codex | Gemini | 置信度 | 处置 |
|---------|--------|-------|--------|--------|------|
| F-01    | ✅     | ✅    | ✅     | 🔴 三命中 | 直接 P0 |
| F-02    | ✅     | ✅    | ❌     | 🟠 高    | P0 候选 |
| F-03    | ✅     | ❌    | ❌     | 🟡 单命中 | 进辩论 |
| F-04    | ❌     | ❌    | ✅     | 🟡 单命中 | 进辩论 |

辩论结论：F-03 被其他两模型反驳 → 不入 P0；F-04 被 Claude 同意 → 升为 P1
```

### 容错

- **部分模型失败**（CLI 挂了 / 超时 / 返回空）：仍用剩余模型出报告，报告开头写明「本次只收到 X/Y 模型的回复」
- **模型输出格式不一致**：不强制统一；主会话做语义对齐即可（「model A 说的『race condition』和 model B 说的『数据竞争』是同一类」）
- **成本爆炸**：跨模型只在**关键变更**（生产 / 架构级 / 安全敏感）启用。默认通道仍是同模型多人格

### 与现有模式的关系

| 模式 | 隔离粒度 | 适用场景 |
|------|---------|---------|
| `review`（默认）| 同模型多 SubAgent 人格隔离 | 所有 PR 默认 |
| `challenge` | 同模型多人格 + 额外挑战者 | 重要 PR，怀疑有盲区 |
| `consult` | 共享上下文 | 小 PR / 向后兼容场景 |
| **cross-model**（本段新增）| 跨模型并行 + 交叉验证 + 辩论 | 生产 / 架构 / 安全敏感变更 |

启用方式：命令行加 `--mode cross-model`，或在 `/lfg` 阶段 5 针对敏感任务自动路由。
