# Team-Spec — 规格制定编队

铁律：**模糊需求不动手。规格不齐不进实施。**

> **场景化预制流水线（吸收自 [Donchitos/Claude-Code-Game-Studios](https://github.com/Donchitos/Claude-Code-Game-Studios) `/team-*` 模式，Issue #55）**：
>
> 当任务进入「需求规格化」阶段时，按本流水线串联 `/spec` → `/plan-check` → `/adr`，避免 orchestrator 临时编排导致漂移。
> 跟 `/lfg` 的关系：`/lfg` 是「需求 → 交付」全流程；`/team-spec` 是只覆盖**规格制定那一段**的局部编队，可独立调用，也可被 `/lfg` 阶段 2.5-3 引用。
> 走 D 方案：4 个 team-* skill 各自独立完整写一遍流水线，不抽共享 base（教训来自 lessons.md 2026-04-27）。

当前项目：`Agent Harness Framework`（cli-tool / python）
测试命令：`make test`

---

## 参数检查

如果 `$ARGUMENTS` 为空，输出：

> 用法：`/team-spec [需求描述]` — 把模糊需求规格化为可测试的验收标准 + 决策记录。

> 示例：`/team-spec 给用户系统加多端登录支持`、`/team-spec 优化首页加载速度`

然后**立即停止**，不进入流水线。

如果有参数，把 `$ARGUMENTS` 作为「需求原始描述」沿着流水线传递。

---

## 适用场景

满足任一即建议走本流水线：

- 需求描述超过两句话
- 预计涉及 3 个以上文件
- 含模糊词：「优化」「改进」「支持」「更好」「灵活」
- 涉及新模块或新架构模式
- 涉及技术选型（语言 / 框架 / 第三方服务）

**不适用**：
- 已有明确验收标准的 bug 修复 → 直接 `/tdd` 或 `/execute-plan`
- 单行配置 / typo 修复 → 直接改

## 编队组成

| 阶段 | 调用 skill | 产出 |
|---|---|---|
| 规格化 | `/spec` | `docs/superpowers/specs/YYYY-MM-DD-<topic>-spec.md`（含 R-ID 列表 + 验收标准） |
| 计划校验 | `/plan-check` | 8 维度结构化校验报告（计划阶段提前发现规格缺口，最多 3 轮修订） |
| 决策记录 | `/adr` | `docs/decisions/<id>-<decision>.md`（仅在涉及架构选择时调用） |

**注意**：`/adr` 是**条件触发**——规格中若出现「技术选型 / 架构模式 / 关键设计权衡」才调用；纯业务需求规格化跳过。

## 流水线

### 阶段 1：规格化

调用 `/spec`，把 `$ARGUMENTS` 作为输入。

`/spec` 完成后产出：
- 验收标准（每条带 R-ID，可机械判定）
- Non-goals（明确不做什么）
- 边界条件 / 错误路径
- 测试策略

**🔴 等用户确认规格**。规格不通过不进下一阶段。

### 阶段 2：计划校验

调用 `/plan-check`，输入是阶段 1 产出的 spec 文档。

`/plan-check` 8 维度校验：
1. 需求覆盖（每条 R-ID 是否有计划步骤映射）
2. 原子性（步骤粒度 2-5 分钟）
3. 依赖排序
4. 文件作用域（每步明确改哪些文件）
5. 可验证性（每步有验证命令）
6. 上下文适配
7. 缺口检测（spec 是否漏了边界）
8. Nyquist 合规（信息密度足够 AI 执行不靠猜）

**最多 3 轮修订循环**：每轮发现缺口 → 回阶段 1 修 spec → 再次校验。3 轮仍不收敛 → **🔴 停下来**告诉用户具体阻塞点。

### 阶段 3：架构决策（条件触发）

读阶段 1 的 spec，扫描是否含以下信号：
- 「选用 X」「采用 Y 模式」「替换为 Z」
- 多个候选方案对比
- 「为什么选 A 不选 B」类讨论

任一命中 → 调用 `/adr` 把决策固化到 `docs/decisions/`，状态默认 `Proposed`。

不命中 → **跳过本阶段**，输出说明「无架构决策需记录」。

### 阶段 4：签收

收集三阶段产物，输出 verdict：

```
## Team-Spec 编队报告

需求：<$ARGUMENTS>

### 阶段产出
- 规格文档：docs/superpowers/specs/<file>.md（R-ID 数：N）
- plan-check 校验：PASS（第 X 轮通过）/ FAIL（阻塞点）
- ADR：docs/decisions/<id>.md（如有）/ 无需记录

### 后续动作建议
- → `/team-implement` 进入实施
- → 或直接 `/write-plan` + `/execute-plan` 单步推进
```

## 错误恢复协议

任一阶段返回 BLOCKED / 错误 / 无法完成时：

1. **立即上报**：在对话里输出 `[<阶段>]: BLOCKED — <理由>`
2. **评估依赖**：下一阶段是否依赖本阶段产出？依赖则停在此处，不强行推进
3. **三选一让用户决策**：
   - **跳过**：标注 gap 在最终报告中，由用户后续手动补
   - **缩范围重试**：把 spec 拆小，只对部分 R-ID 走完整流水线
   - **停下解决**：先解决阻塞再继续
4. **始终产出部分报告**：哪怕只完成阶段 1 也要输出阶段 1 产物，绝不丢弃已完成工作

常见阻塞：
- 阶段 1：需求过于模糊，连 R-ID 都列不出 → 回到 `/brainstorm` 或 `/ideate` 探索方向
- 阶段 2：plan-check 3 轮未收敛 → spec 本身有结构性问题，需重写
- 阶段 3：架构决策需用户裁决（多个方案势均力敌）→ 等用户选

---

## 文件写入约定

- **本编队不直接写代码**——只产出规格 / 校验报告 / ADR 三类**纯文档**
- 每个被调用的 sub-skill（`/spec` / `/plan-check` / `/adr`）自己负责文件写入；本编队只串联调度
- 写入路径白名单：
  - `docs/superpowers/specs/`（规格）
  - `docs/decisions/`（ADR）
  - `.agent-harness/current-task.md`（进度更新）
- 禁止写入：`src/`、`tests/`、`.claude/`（不是规格阶段该碰的位置）

---

## 输出

**verdict 三态**：
- **COMPLETE** — 三阶段都跑完，规格 + 校验 + 决策齐备
- **PARTIAL** — 部分阶段完成（含 gap 说明）
- **BLOCKED** — 在阶段 X 阻塞，需用户介入
