# 工程回顾

核心原则：**用数据说话，用具体事实表扬，用可执行建议改进。**

当前项目：`Agent Harness Framework`（cli-tool / python）

## 第 1 步：数据采集

运行以下命令，收集过去 7 天的工程数据：

```bash
# 提交总览
git log --oneline --since="7 days ago"

# 贡献者分布
git shortlog -sn --since="7 days ago"

# 代码行变更
git log --stat --since="7 days ago"

# 提交时间线（用于工作会话检测）
git log --format="%H %ai" --since="7 days ago"
```

将所有输出保存为工作变量，后续步骤需要用到。

如果提交数为零，直接告知用户"本周暂无提交记录"，终止流程。

## 第 2 步：指标计算

从采集的数据中提取以下指标：

| 指标 | 计算方式 |
|---|---|
| 总提交数 | `git log --oneline` 行数 |
| 新增行数 | `--stat` 输出中 insertions 总和 |
| 删除行数 | `--stat` 输出中 deletions 总和 |
| 贡献者数 | `git shortlog -sn` 行数 |

### 提交类型分布

统计 commit message 前缀，分为以下类别：
- `feat:` — 新功能
- `fix:` — 修复
- `refactor:` — 重构
- `test:` — 测试
- `chore:` — 杂务
- `docs:` — 文档
- 其他 — 未分类

### 关键比率

- **修复比率** = fix 提交数 / 总提交数
  - > 40% 可能表示代码稳定性问题，需要关注
  - < 20% 表示健康状态
- **测试比率** = test 提交数 / 总提交数
  - > 20% 表示测试意识良好
  - < 5% 建议加强测试习惯

## 第 3 步：工作会话检测

将提交按时间排序，根据 **45 分钟间隔** 分组为独立会话。

### 会话分类

| 类型 | 持续时间 | 说明 |
|---|---|---|
| 深度会话 | > 2 小时 | 专注的大块工作时间 |
| 中等会话 | 30 分钟 – 2 小时 | 常规开发节奏 |
| 微会话 | < 30 分钟 | 快速修复或小调整 |

报告各类会话的数量和占比。深度会话占比低于 30% 时，建议改善专注时间管理。

## 第 4 步：热点分析

### 高频变更文件

统计被修改次数最多的前 10 个文件：
- 修改次数 >= 5 的文件标记为**复杂度热点**，可能需要拆分或重构
- 查看这些文件是否集中在某个模块，判断是否存在架构瓶颈

### 修复热点

筛选仅出现在 `fix:` 提交中的文件：
- 某文件频繁出现在修复提交中，标记为**质量风险点**
- 建议对这些文件增加测试覆盖

## 第 5 步：贡献者分析

对每位贡献者生成独立分析：

### 分析维度

1. **提交数量**：本周提交总数
2. **关注领域**：根据修改的文件路径推断（如 `src/api/` → API 层）
3. **提交类型分布**：feat / fix / refactor / test / chore / docs 各多少
4. **具体表扬**：锚定到实际提交
   - 好的例子："重构了认证模块（commit abc123），将复杂度降低了 40%"
   - 坏的例子："代码写得不错"（太笼统，禁止使用）
5. **成长建议**：基于提交模式给出具体建议
   - 如果 fix 占比过高："建议在开发新功能时先写测试，减少后续修复"
   - 如果缺少 test 提交："建议尝试 TDD 流程，每个 feat 配套至少一个 test"

## 第 6 步：输出报告

按以下结构生成最终报告：

### 1. 一句话总结

用一句话概括本周状态（适合发 Slack/飞书），不超过 50 字。

### 2. 指标表格

展示第 2 步计算的所有指标，使用 Markdown 表格。

### 3. 本周 Top 3 亮点

列出 3 件做得好的具体事情，每条附带提交引用：
- 必须有 commit hash 或 PR 编号
- 必须说明为什么这件事值得表扬

### 4. Top 3 改进方向

列出 3 个具体、可执行的改进建议：
- 每条建议必须包含"做什么"和"为什么"
- 避免空泛建议（如"提高代码质量"）

### 5. 下周习惯建议

根据本周数据推荐 2-3 个可立即执行的习惯调整。

## 任务日志集成

检查 `.agent-harness/task-log.md` 是否存在：
- 如果存在，读取已完成任务列表，与提交记录交叉引用
- 将任务完成情况纳入亮点或改进方向的分析中
- 如果不存在，跳过此步骤，不报错

## 注意事项

- 所有数据基于 `Agent Harness Framework` 仓库的 git 历史
- 不要编造数据，如果某个指标无法计算，明确说明原因
- 保持客观：表扬要具体，批评要建设性
- 如果只有一位贡献者，跳过"贡献者分析"章节的对比部分
