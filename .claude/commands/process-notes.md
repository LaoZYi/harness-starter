# 处理产品需求笔记

你的任务是读取 `notes/` 目录下的需求笔记，从中提取结构化信息，填充到 harness 工程文档中。

## 背景

这些笔记是用户通过语音转文字记录的产品需求描述。内容特点：
- 按思维流组织，不是按文档结构组织
- 可能有口语化表达、重复、跳跃
- 同一个话题可能分散在多处提到
- 核心价值是**用户的意图和决策**，而非措辞本身

当前项目：`Agent Harness Framework`（cli-tool）

## 执行步骤

### 第 1 步：读取笔记

扫描 `notes/` 目录，读取所有 `.md` 和 `.txt` 文件。如果目录为空或不存在，告知用户并停止。

### 第 2 步：读取当前文档

读取以下 harness 文档的当前内容：
- `docs/product.md`
- `docs/architecture.md`
- `docs/runbook.md`
- `docs/workflow.md`

### 第 3 步：提取与映射

从笔记中提取信息，按以下规则映射到目标文档：

| 笔记中的内容类型 | 目标文档 | 目标位置 |
|---|---|---|
| 产品定位、一句话描述 | `docs/product.md` | 「一句话目标」 |
| 产品的详细说明、背景、动机 | `docs/product.md` | 「详细描述」 |
| 要做的功能、特性 | `docs/product.md` | 「功能列表」 |
| 验收标准、怎样算做完 | `docs/product.md` | 「完成标准」 |
| 技术选型、要用的服务/数据库/API | `docs/architecture.md` | 「外部系统」 |
| 架构决策、模块划分 | `docs/architecture.md` | 「架构整理建议」 |
| 怎么运行、部署、配置 | `docs/runbook.md` | 对应章节 |
| 开发流程、协作约束 | `docs/workflow.md` | 「额外提示」 |

提取原则：
- **合并同类项**：散落在笔记各处的同一话题要归到一起
- **提炼意图**：口语化表达转化为清晰简洁的文档语言
- **保留决策**：如果用户明确说了"用 X 不用 Y"，保留这个决策及原因
- **忽略噪音**：语气词、重复、无信息量的口头禅直接过滤
- **存疑标注**：含糊不清或矛盾的内容用 `<!-- 待确认：... -->` 标注

### 第 4 步：展示摘要

**在写入任何文件之前**，先向用户展示结构化摘要。格式如下：

```
## 提取摘要

### docs/product.md
- **一句话目标**：...
- **详细描述**：...
- **功能列表**：
  - [ ] 功能 1：描述
  - [ ] 功能 2：描述
  ...
- **完成标准**：
  - [ ] 标准 1
  - [ ] 标准 2
  ...

### docs/architecture.md
- **外部系统**：
  - 系统 1：用途
  ...
- **架构建议**：...

### docs/runbook.md
- （如有运行/部署相关内容）

### docs/workflow.md
- （如有流程约束相关内容）

### 未归类内容
- （笔记中提到但不属于以上任何类别的内容）

### 待确认项
- （含糊或矛盾的内容）
```

然后询问用户是否确认写入，或需要调整。

### 第 5 步：写入文档

用户确认后，将提取的内容写入对应文档：

- 替换现有的占位符文本（如"待补充"）
- 保留文档的 markdown 结构和注释标记
- 功能列表使用 `- [ ]` checkbox 格式
- 完成标准使用 `- [ ]` checkbox 格式
- 不要覆盖文档中已有的非占位符内容，而是补充或合并

### 第 6 步：标记已处理

在每个已处理的笔记文件头部添加：

```
<!-- processed: YYYY-MM-DD -->
```

这样后续再运行此命令时可以跳过已处理的文件（除非用户要求重新处理）。

## 注意事项

- 如果某个文档类别没有从笔记中提取到内容，**不要填充**，保持原样
- 如果笔记内容与文档现有内容冲突，在摘要中明确指出让用户决定
- 写入后运行 `python scripts/check_repo.py` 验证项目状态
