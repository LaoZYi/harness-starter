# 测试驱动开发（TDD）

铁律：**没有失败的测试，就不写生产代码。**

当前项目：`Agent Harness Framework`（cli-tool / python）
测试命令：`make test`

## 适用场景

以下情况**必须**使用 TDD 流程：
- 新增功能
- 修复 Bug
- 重构（行为变更时）
- 修改任何影响外部可观察行为的代码

例外情况（需要用户明确批准）：
- 纯文档变更
- 配置文件调整（无逻辑）
- 依赖版本升级（无 API 变更）

## RED-GREEN-REFACTOR 循环

### 第 1 步：RED — 写失败测试

写一个**最小的**测试，描述你期望的行为：
- 测试名称要说明被测行为，不是实现细节
- 一个测试只验证一个行为
- 使用项目已有的测试框架和风格

```
好的测试名：test_user_login_returns_token_on_valid_credentials
坏的测试名：test_login_function
```

### 第 2 步：验证 RED

运行测试：
```
make test
```

确认测试**失败了**，并且失败原因是**正确的**：
- 如果报编译错误 → 正常，说明目标代码还不存在
- 如果报断言失败 → 正常，说明行为尚未实现
- 如果测试通过了 → **停下来**，你的测试没有测到新行为
- 如果报无关错误 → 修复测试本身，不要继续

### 第 3 步：GREEN — 最简实现

写**刚好能让测试通过**的代码：
- 不要提前优化
- 不要添加测试未要求的功能
- 硬编码返回值也可以（后续测试会逼出真实逻辑）

### 第 4 步：验证 GREEN

再次运行：
```
make test
```

确认：
- [ ] 新测试通过
- [ ] 所有已有测试仍然通过（无回归）
- [ ] 没有跳过的测试

如果有测试失败：
- 先判断是新代码引起的还是既有问题
- 修复后再次运行全部测试
- **绝对不要跳过或注释掉失败的测试**

### 第 5 步：REFACTOR — 清理

在所有测试通过的状态下重构：
- 消除重复
- 改善命名
- 提取方法 / 常量
- 简化逻辑

重构规则：
- 每次小改动后运行 `make test`
- 保持测试绿色
- 不改变外部行为

## 循环节奏

```
RED → 验证失败 → GREEN → 验证通过 → REFACTOR → 验证通过 → 提交 → 下一个 RED
```

每完成一个 RED-GREEN-REFACTOR 循环后做一次原子 commit。

## 禁止事项

- 禁止在测试通过前写下一个功能的代码
- 禁止一次写多个测试然后一起实现
- 禁止删除或跳过失败的测试来"修复"问题
- 禁止在 REFACTOR 阶段添加新行为
- 禁止写测试时假设实现细节（测试接口，不测内部）

## 完成检查清单

一组功能实现完毕后：
- [ ] 所有测试通过：`make test`
- [ ] 测试覆盖了正常路径和错误路径
- [ ] 没有被跳过或注释掉的测试
- [ ] 每个 commit 对应一个 RED-GREEN-REFACTOR 循环
- [ ] 代码无明显重复
