# 测试驱动开发（TDD）

铁律：**没有失败的测试，就不写生产代码。**

当前项目：`Agent Harness Framework`（cli-tool / python）
测试命令：`make test`

## 适用场景

以下情况**必须**使用 TDD 流程：
- 新增功能
- 修复 Bug
- 重构（行为变更时）
- 修改任何影响外部可观察行为的代码

例外情况（需要用户明确批准）：
- 纯文档变更
- 配置文件调整（无逻辑）
- 依赖版本升级（无 API 变更）

## RED-GREEN-REFACTOR 循环

### 第 1 步：RED — 写失败测试

写一个**最小的**测试，描述你期望的行为：
- 测试名称要说明被测行为，不是实现细节
- 一个测试只验证一个行为
- 使用项目已有的测试框架和风格

```
好的测试名：test_user_login_returns_token_on_valid_credentials
坏的测试名：test_login_function
```

### 第 2 步：验证 RED

运行测试：
```
make test
```

确认测试**失败了**，并且失败原因是**正确的**：
- 如果报编译错误 → 正常，说明目标代码还不存在
- 如果报断言失败 → 正常，说明行为尚未实现
- 如果测试通过了 → **停下来**，你的测试没有测到新行为
- 如果报无关错误 → 修复测试本身，不要继续

### 第 3 步：GREEN — 最简实现

写**刚好能让测试通过**的代码：
- 不要提前优化
- 不要添加测试未要求的功能
- 硬编码返回值也可以（后续测试会逼出真实逻辑）

### 第 4 步：验证 GREEN

再次运行：
```
make test
```

确认：
- [ ] 新测试通过
- [ ] 所有已有测试仍然通过（无回归）
- [ ] 没有跳过的测试

如果有测试失败：
- 先判断是新代码引起的还是既有问题
- 修复后再次运行全部测试
- **绝对不要跳过或注释掉失败的测试**

### 第 5 步：REFACTOR — 清理

在所有测试通过的状态下重构：
- 消除重复
- 改善命名
- 提取方法 / 常量
- 简化逻辑

重构规则：
- 每次小改动后运行 `make test`
- 保持测试绿色
- 不改变外部行为

### 第 5.5 步：测试质量增强（可选，Issue #41）

在 REFACTOR 完成后，评估测试套件本身的有效性——测试全绿 ≠ 测试够好。

#### Mutation Testing（突变测试）

对刚写的实现代码做微小突变（改 `>` 为 `>=`、删条件分支、替换返回值），重跑测试：
- 测试能捕获突变 → 突变被杀死（好）
- 测试仍通过 → 突变存活（测试不够强）

```
突变测试结果：注入 N 个突变，杀死 M 个，存活 K 个
突变存活率：K/N = XX%
```

突变存活率 > 30% 时建议补充测试用例覆盖存活的突变点。

#### Property-Based Testing（属性测试）

对纯函数和数据转换，除 example-based 测试外，考虑补充属性测试：
- 验证不变量（如「排序后长度不变」「序列化-反序列化幂等」）
- 用随机输入验证边界行为
- 适用场景：数据转换、编解码、数学运算、集合操作

> 此步骤为**建议性**，不阻塞循环。在关键业务逻辑（支付、权限、数据迁移）中强烈推荐执行。

## 循环节奏

```
RED → 验证失败 → GREEN → 验证通过 → REFACTOR → 验证通过 → 提交 → 下一个 RED
```

每完成一个 RED-GREEN-REFACTOR 循环后做一次原子 commit。

## 禁止事项

- 禁止在测试通过前写下一个功能的代码
- 禁止一次写多个测试然后一起实现
- 禁止删除或跳过失败的测试来"修复"问题
- 禁止在 REFACTOR 阶段添加新行为
- 禁止写测试时假设实现细节（测试接口，不测内部）

## 反合理化

当你想跳过 TDD 流程时，检查这张表。这些都是 agent 常见的自我说服理由——每一条都有反驳。

| 你想说的 | 为什么不行 |
|---------|-----------|
| "这个改动太小，不需要测试" | 越小的改动越应该有测试——写测试的成本也很小，而缺少测试的小改动是回归 bug 的温床 |
| "我先把功能写完再补测试" | 那不是 TDD，那是"测试后补"。先写测试的目的是用测试驱动设计，事后补测试只是走形式 |
| "这段代码很简单，不可能出错" | 生产事故报告里出现最多的一句话就是"这段代码很简单"。简单的代码更容易测试，没有理由跳过 |
| "时间不够了，先上线再说" | 没有测试的代码上线后出 bug 的修复时间远大于写测试的时间。技术债不会自动消失 |
| "这只是内部工具/脚本" | 内部工具出 bug 时没有用户反馈，问题会更隐蔽。测试是你唯一的安全网 |
| "重构不改行为，不需要新测试" | 重构不需要新测试，但必须在已有测试保护下进行。如果没有测试覆盖，先补测试再重构 |

**铁律不变：没有失败的测试，就不写生产代码。**

## 完成检查清单

一组功能实现完毕后：
- [ ] 所有测试通过：`make test`
- [ ] 测试覆盖了正常路径和错误路径
- [ ] 没有被跳过或注释掉的测试
- [ ] 每个 commit 对应一个 RED-GREEN-REFACTOR 循环
- [ ] 代码无明显重复

## StuckDetector 接入

TDD 循环中**连续 3 次 RED 未进展**（同一测试、同一报错、同一根因假设）→ 必须触发 task-lifecycle 规则中的 StuckDetector：立即停下、写卡点记录、输出 3 个候选方向给用户选。禁止在沉没成本上继续投 tokens。详见 `.claude/rules/task-lifecycle.md` "卡死检测"一节。
