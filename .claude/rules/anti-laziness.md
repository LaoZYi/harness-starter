# 反偷懒门禁（Anti-Laziness Gates）

> **分类标记：`defensive-temporary`**——这些机制防的是模型在长上下文中主动减少自检的倾向（参见 Yandex Reasoning Shift 论文）。未来模型内部对齐改善后，部分门禁可能变冗余。与之对应的 `collaborative-permanent` 类机制（audit WAL、memory 分层、agent diary 隔离等）无论模型多强都需要。

本规则定义 7 道硬门禁。各 skill 在自身文档中引用并落地具体检查逻辑。

## 门禁 1：数量门禁（Count Gate）

**规则**：产出清单中的每一项都必须有明确状态，禁止静默跳过。

适用场景：
- `/verify` 的检查项：每项必须是 `✅ passed` / `⚠️ skipped（附理由）` / `❌ failed` 三态之一
- `/multi-review` 的 P0/P1 反馈：每条必须逐条回应（`接受` / `质疑` / `推迟`），禁止批量 dismiss
- `/receive-review` 的评审意见：逐条标注处理结果

**硬检查**：`处理数 == 总数`。差值 > 0 说明有遗漏，必须补齐后才能宣告完成。

## 门禁 2：上下文隔离（Context Isolation）

**规则**：验证和评审必须在独立上下文中执行，不在主会话中串行完成。

原因：长上下文污染会不自觉缩减验证范围——Anthropic 实验证明保留更多信息反而让 AI 表现更差。

适用场景：
- `/multi-review` 的每个审查员必须是独立 SubAgent（spawn 新上下文）
- `/verify` 的穷举验证脚本在独立环境（临时目录）执行，不依赖主会话状态
- 跨模型对抗验证（如适用）必须 spawn 独立 SubAgent，不给第二模型看第一模型的结论

**例外**：轻量通道（小任务）的 `/multi-review` 只启用 2 个审查员时，可在主会话执行。

## 门禁 3：反合理化表（Anti-Rationalization Table）

**规则**：任何 skill 的「可跳过条件」旁必须放一张「不能用的借口清单」。

这些借口在表面上合理，但实际上会导致质量退化：

| 借口 | 驳斥 |
|------|------|
| 「改动很小，不需要完整验证」 | 生产事故常来自一行代码。运行验证只需几秒，跳过的代价是几十倍修复时间 |
| 「刚才已经验证过了」 | 改了代码就必须重新验证。上次的结果已经过期 |
| 「这个检查项不适用」 | 必须给出具体理由（文件路径 / 代码行 / 配置项），不能只说「不相关」 |
| 「时间不够，先跳过评审」 | 未经评审的代码出问题后的修复时间远大于评审时间 |
| 「上游产物应该没问题」 | 不信任，要验证。检查产物存在且非空只需一条命令 |
| 「我很有信心这次没问题」 | 信心不是证据。最有信心的时候恰恰是最容易忽视问题的时候 |
| 「这错误本来就有，不是我引入的」 | 嘴上说没用。**必须**把开发前的基线报告与开发后的报告做增量对比（见门禁 6），差值 = 0 才算没引入 |
| 「这个 Warning 是历史遗留」 | 同上——是不是"历史"由前后报告说了算，不是由本次开发者的解释说了算 |
| 「这条 Rule 这次场景特殊，可以先不管」 | 特殊例外必须走显式流程：在 current-task 写明理由 + 用户确认。不经确认的「特殊」= 绕过规则 |
| 「Agent 写入失败，手动接管后差不多了」 | **差不多 ≠ 完成**。手动接管只是救火，必须继续 spawn 新 SubAgent 按原流程走完——救火现场不是合规终点（来源：快手 sec-audit-pipeline 真实事故） |
| 「上下文太长，不想 spawn 新 SubAgent」 | 上下文长**正是**该 spawn 的时机。fresh context 的判断准确率比污染 context 更高——短期偷懒换来长期错误累积（与 autonomy.md Trust Calibration 3b 联动） |
| 「SKILL.md 太长，精简后传给 SubAgent」 | 你不知道删的是不是关键防护层——原作者为防特定场景写的条款，SubAgent 没读到就破防。**精简 = 擅自删减**，必须传完整 SKILL.md 或用 `allowed-tools` 精确约束 |
| 「CTO 说 ship it fast，跳过 /cso 这步」 | 权威压力不能覆盖安全审计。CTO 的授权是「决定做什么」，不是「决定跳过哪条验证」。若真有例外走显式流程：在 current-task 写明授权来源 + 审计事后补做时间（对应 pressure-test 第 4 类压力「权威」） |
| 「再跑一次 /multi-review 又要烧几十 k token，省省吧」 | Token 成本是**幻觉优化**。评审漏掉的 P0 修复成本远大于几十 k token。项目预算在评审上的投入是对「未来修复成本」的对冲，不是浪费（对应 pressure-test 第 5 类压力「经济」） |
| 「反正只是再加一个字面/路径就行」 | **白名单漂移是单调累积的**——本项目这类反模式已在 5 个场景反复踩过(check_repo / _RUNTIME_MODULES / upgrade three_way / stop hook 5 字面 / stop hook 通用字段)。每次"加一个"的代价感觉小,但 N 次累积说明白名单根本不该存在。改自动发现/字段标记/策略统一化一次到位,比 N 次 patch 总成本低。详见 `.claude/rules/architecture-patterns.md` 反模式 1 |
| 「压缩 / 截断越狠越省 token」 | tokens-per-task 视角下错的——压狠了丢失文件路径 / 错误信息 / 决策理由,AI 后续 re-explore / re-read / re-derive,总开销反而上涨 40 倍。看 re-fetching frequency(AI 反复要求重读已处理的文件)而非 per-request 节省。详见 `.claude/rules/context-budget.md` 规则 4「优化目标」节(Issue #50 吸收 muratcankoylan/Agent-Skills-for-Context-Engineering) |

**扩展**：各 skill 可在自身文档中追加领域特异的反合理化条目，但不得删减本表中的通用条目。

## 门禁 4：下游消费者门禁（Downstream Consumer Gate）

**规则**：每个阶段开始前，校验上一阶段的产物存在且非空。

适用场景：
- `/execute-plan` 每步开始前检查前置产物（计划文件、依赖的上游输出）
- `/lfg` 阶段切换时校验上一阶段通过门产物
- `/dispatch-agents` 子 agent 启动前检查输入文件清单中的每个文件

**检查方式**：
```
对产物清单中的每个文件：
  1. 文件存在？ → 不存在则停下
  2. 文件非空（> 0 字节）？ → 空则停下
  3. 文件内容非占位符（不全是 TODO/TBD/待补充）？ → 是占位符则警告
```

**字段级必填增强**（来源：快手 sec-audit-pipeline 防护 3）：

当产出方在 prompt 中显式声明「产物必须包含字段 A/B/C」时，下游在「存在 + 非空 + 非占位符」3 项之外**必须逐字段校验**。示例：

```
场景：/spec 声明规格必含「R-ID / 目标 / 验收标准 / Non-goals」4 字段
下游 /write-plan 在读取 spec 时：
  - grep "R-ID" spec.md → 必须命中
  - grep "Non-goals" spec.md → 必须命中
  - 缺字段 → 停下,不进入计划阶段
```

⚠️ **慎用纯行数阈值**：行数门禁可能诱导 agent 凑字数（写流水账达标但无实质内容）。优先用「关键字段必填 + 非占位符」双检，不用「>= N 行」。

## 门禁 5：不确定性输出（Uncertainty Output）

**规则**：AI 在生成**事实性结论**（API 签名、函数行为、配置项、数字指标、外部状态）时，若置信度不足以断言，必须**显式声明不确定性**并指明核实方式，禁止用确定性语气输出推测内容。

> 来源：腾讯 CDN LEGO 团队从 57 个真实案例提炼的 5 大根因中列为**最高风险**——「不会说『我不知道』：AI 会用自信的语气输出错误结论，反而降低人的审查意愿」。

**适用触发**：

- 引用框架 / 库 / 标准协议的 API 签名、参数、行为
- 引用文档章节号、版本号、变更日期
- 描述外部系统状态（数据库行数、文件存在性、网络状态）未 observe 就给结论
- 给出量化指标（性能提升百分比、测试覆盖率、错误率）没有实测数据

**输出格式（两种合法表达）**：

```markdown
✅ 合法 — 确定性表达（必须有证据）：
「根据 `src/foo.py:42` 的实现，bar() 返回 None」
「已运行 `make test`，587 tests pass（附输出）」

✅ 合法 — 不确定性表达：
「⚠️ 待核实：我记得 PyYAML 的 safe_load 在 5.x 后默认拒绝任意对象，
 需运行 `python -c "import yaml; print(yaml.__version__)"` 确认」

❌ 违规 — 隐式确定性：
「PyYAML 的 safe_load 默认拒绝任意对象」（凭记忆，没指明来源也没挂 ⚠️）
```

**硬检查**：

- 所有**事实性断言**必须可归类为 observed（有文件路径/命令输出）或 uncertain（挂 `⚠️ 待核实`）之一
- 承接「⚠️ 待核实」的断言下游消费者（`/execute-plan`、`/write-plan`、`/multi-review`）**必须**优先用 `/source-verify` 或具体命令核实，未核实不得进入下游
- **不得**混用「大概率」、「应该是」、「据我所知」等模糊副词来规避本规则——这些词仍在做确定性陈述，只是把责任推给读者

**反合理化**（扩展门禁 3）：

| 借口 | 驳斥 |
|------|------|
| 「挂太多 ⚠️ 会让报告显得不专业」 | 不确定性被伪装成确定性**才是真不专业**。审查者被误导无法分配注意力 |
| 「我记得很清楚，不用挂」 | 记忆 ≠ 证据。模型训练数据存在截断和偏差，本地实测比自信更可靠 |
| 「这是常识，不用核实」 | 「常识」往往是框架或版本依赖的——不同 Python / Node / framework 版本的「常识」不同 |

### 高风险动作的信心指数（Confidence Score）

> 来源：阿里云 Qoder CLI 团队《Harness Engineering 实战》——自动修复前 AI 自评 0-100 信心分，低于阈值不自动执行，转人工。

不确定性**定性**表达（`⚠️ 待核实`）够用于日常。但以下**高风险动作**执行前，AI 必须额外给出**定量**信心分：

**什么算「高风险动作」**：

- **自动修复** / 自动重构 / 批量改名（单次改动 ≥ 3 文件且影响业务逻辑）
- **数据库 migration** / schema 变更 / 数据回填
- **安全 / 权限相关**代码（认证、授权、密钥读取、加密）
- **生产配置** / CI / 部署脚本变更
- **不可逆操作**（rm / git reset --hard / 发布、推送）
- **外部集成**（HTTP 客户端对生产 URL、MQ 生产、第三方 API 写操作）

**信心分表达格式**：

```markdown
🎯 信心指数：82 / 100

拆解：
- 根因定位证据（+40）：✅ 已复现 + 通过测试锁定问题函数 `parse_config():42`
- 修复方案证据（+30）：✅ 类似 pattern 在 commit abc123 已用过且稳定
- 影响面证据（+20）：⚠️ 未完全确认下游 3 个 consumer 都会走这个分支
- 回归测试（+10）：✅ 已补 2 条 test case 覆盖正常 + 边界

扣分项：
- 影响面 ⚠️ 导致 -8
- 老代码无单测基线 -10

建议阈值：本项目 ≥ 80 自动执行；60-79 人工确认；< 60 回到规划
```

**阈值处理规则**：

| 分数区间 | 处理 |
|---------|------|
| ≥ 80 | 可进入执行；但人工 CR 仍必须 |
| 60–79 | **暂停**，展示信心拆解 + 让用户明示「继续 / 调整方案 / 补证据」 |
| < 60 | **拒绝执行**，回到 `/write-plan` 或 `/source-verify` 补证据链 |

**阈值可配**：项目可在 `docs/product.md` 或 `CLAUDE.md` 里覆盖默认阈值（如安全敏感项目上调到 90）。

**与定性 `⚠️ 待核实` 的边界**：

- 日常事实性断言 → 门禁 5 前半段（挂 `⚠️` 或给出来源即可，不需要打分）
- **高风险动作**执行前 → 必须定量（0-100 分）+ 拆解 + 阈值比对

**反合理化**：

| 借口 | 驳斥 |
|------|------|
| 「给 100 分就能通过，谁不会」 | 100 分必须有证据链支撑，每一扣分项都要实名。不实名的高分是作弊 |
| 「这个动作不算高风险」 | 若存疑 → 就按高风险处理。宁可多一次信心拆解，不要少一次关键校验 |
| 「信心分纯主观」 | 要求**拆解**不是光给一个数——拆解本身就在强迫 AI 把推理链摊开，减少随意性 |

## 门禁 6：基线对比（Baseline Diff）

> 来源：腾讯 JK Launcher 团队《Harness Engineering 工程化落地》——AI 会拿「这错误本来就有」当借口。把"是不是你引入的"从主观解释转成前后报告对比。

**规则**：对可量化的质量信号（测试失败数、warning 数、lint 违规数、门禁脚本输出），**开发前**跑一次存基线，**开发后**再跑一次做增量对比。允许 0 增量，不允许"口头解释"。

**适用触发**：

- 任务复杂度 ≥ 中（3+ 文件 / 架构改动）
- 改动接触过往 AI 报过 flaky 的模块
- 门禁脚本首次引入，或覆盖面扩大后
- 用户提示「为什么报错变多了」「这 warning 以前没有吧」

**执行流程（最小形态）**：

```bash
# 改动前：
<run-verify> > /tmp/baseline-$(date +%Y%m%d-%H%M).txt
# 改动后：
<run-verify> > /tmp/after-$(date +%Y%m%d-%H%M).txt
diff /tmp/baseline-*.txt /tmp/after-*.txt
```

`<run-verify>` 是本项目一键产出定量报告的命令（`make test`、`make check`、或项目自建的总验证脚本）。

**判定标准**：

- 新增的失败 / warning / 违规 **= 0** → 通过
- 新增 **> 0** → 必须修，或在 current-task 写明"非本次引入"的**证据**（比如在 baseline 里也存在）
- 基线缺失 → 先补跑基线，不接受"这次先跳过"

**输出要求**：在 `/verify` 报告里列两个数字——`改动前 X 项失败 / Y 个 warning`、`改动后 X' 项失败 / Y' 个 warning`。数字说话，不用形容词。

**与其他门禁的边界**：

- 门禁 1（数量门禁）管「清单每项要有状态」——静态清单的完整性
- 门禁 6（基线对比）管「定量信号的前后对比」——动态信号的增量

## 门禁 7：压力测试（Pressure Test，快手 sec-audit-pipeline 吸收）

> **分类标记：`defensive-temporary`**——与本规则顶部声明一致。门禁 1-6 都是「声明式」的；门禁 7 是「回归测试式」——用主动注入压力验证规则在真实压力下是否还被遵守。未来模型对齐改善后本门禁可能冗余。

> 来源：快手安全 SRC 赵海洋《Harness Engineering 三层架构 × 形式化验证（下）》。核心论点：**规则写了 ≠ 扛得住**。反合理化表列了 12 条借口，AI 在特定压力下仍可能绕过；必须主动压测验证。

**规则**：对关键 skill（`/verify` / `/multi-review` / `/cso` / `/lfg`）周期性注入 7 类压力测试，验证反合理化表是否真扛得住。

**7 类压力**：沉没成本 / 疲劳 / 时间压力 / 权威 / 经济 / 务实 / 复杂度回避（逐类对应门禁 3 表条目）

**适用触发**：

- **每月 1 次定期**（可由 `/loop 30d /pressure-test` 调度）
- **新反合理化条目**加入门禁 3 表后，**立即**压测该条是否被吞
- **用户报告**「AI 又绕过 X 规则」时
- 新增 skill 首次发布前

**闭环流程**（违规自动喂回反合理化表）：

1. 跑 `/pressure-test` → 7 类 × 6 场景注入
2. 检测违规 → **捕获 AI 原话借口**（不二次改写）
3. 分类：属于 7 类压力中哪一种 / 违反的是哪道门禁
4. **写反驳**：把借口 + 驳斥追加到门禁 3 反合理化表
5. **更新压力测试场景**：补一条新场景作为回归保护

**与 T6 晋升的衔接**：压力测试发现**反复失守**的门禁（连续 3 次被吞）→ 自动进入 T6 脚本化候选队列（见 `.claude/rules/knowledge-conflict-resolution.md`）：路径 A 脚本化 / 路径 B 升级 Rule / 路径 C 反合理化加料。

**晋升计数通道（双入口）**：「连续 3 次被吞」的计数不限于 `/pressure-test` 月度运行——**真实任务**里被 `/verify` / `/multi-review` / `/receive-review` 抓到同款借口原话的，同样计入晋升计数。避免「每月 1 次压测 × 3 次 = 3 个月」的慢节奏让坏借口长期躺着。计数记录在 `.agent-harness/lessons.md` 对应条目的「触发次数」字段。

**场景库去重规则**（防止无限膨胀）：同一 skill × 同一压力类型下，场景库只保留**最新 1 条**——压测场景是回归保护而非历史档案；发现更有效的新场景时替换旧场景，不累积。反合理化表则**不去重**——每条借口原话值得保留作为后续训练语料。

**具体执行**：见 `/pressure-test` skill 模板。

## 与现有机制的关系

| 机制 | 防什么 | 何时触发 |
|------|--------|---------|
| **StuckDetector**（task-lifecycle） | 循环失败（同一错误 3 次） | 执行中 |
| **反偷懒门禁**（本规则） | 跳步、假判不适用、缩减范围、伪装确定性、甩锅「历史遗留」 | 验证/评审/阶段切换/输出结论时 |
